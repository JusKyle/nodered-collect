# 技术栈选型

## 1. 核心决策 (Key Decisions)
*简述为什么选择这套技术栈？*

*   **选择 TypeScript + Node.js**：前后端语言统一，降低开发和维护成本；Node.js 异步 I/O 天然适合 MQTT + HTTP + SSE 的多协议网关管理场景。
*   **选择 React + Vite**：React 生态成熟，组件化适合管理网关列表、实例详情等多页面；Vite 开发速度快，构建体积小。
*   **选择 EMQX 作为 MQTT Broker**：工业级高可用 Broker，支持大规模设备连接，天然契合边缘网关 → 平台 的数据上报模型。
*   **选择 PostgreSQL + Prisma**：Prisma 让数据库 Schema 演进可控，PostgreSQL 的 JSON 类型支持灵活存储模板配置和点位字段。
*   **选择 Redis**：心跳时间戳、下发锁等高频读写需要内存级缓存，Redis 天然胜任。
*   **选择 Node-RED 作为网关执行环境**：工业场景已广泛使用，节点生态丰富（modbus/S7/OPC UA 等协议节点成熟），通过 `/flows` API 可远程动态部署采集逻辑。
*   **选择 SQLite 作为网关侧缓存存储**：零配置嵌入式数据库，Node-RED 插件直接调用，适合本地缓存场景，无需在网关侧部署额外服务。

## 2. 前端 (Frontend)
*   **框架**: React 18 + React Router 6 (函数组件 + Hooks)
*   **样式方案**: TailwindCSS + PostCSS + autoprefixer
*   **状态管理**: Zustand 4.x（轻量、Store 分层，分别管理网关、模板、实例等状态）
*   **构建工具**: Vite 5.0
*   **数据交互**: axios 1.x（HTTP 请求）+ 浏览器原生 EventSource（SSE 实时推送）
*   **表单验证**: react-hook-form + @hookform/resolvers + zod 3.x
*   **图表**: recharts 2.x（网关性能趋势图、采集值历史图）
*   **开发工具**: TypeScript 5.2 + ESLint + @typescript-eslint

## 3. 后端 (Backend)
*   **运行时/语言**: Node.js 18+ / TypeScript 5.2
*   **Web 框架**: Express 4.18（RESTful API 服务）
*   **API 协议**: REST（JSON over HTTP）+ Server-Sent Events（SSE，用于实时状态推送）
*   **核心依赖**:
    *   `@prisma/client` 5.x：ORM 数据访问层
    *   `mqtt` 5.x：MQTT 客户端，订阅网关数据上报
    *   `redis` 4.x：Redis 客户端，管理心跳/Token/下发锁
    *   `axios` 1.x：调用 Node-RED `/flows` API 下发配置
    *   `zod` 3.x：API 请求参数验证
    *   `jsonwebtoken` 9.x + `bcrypt` 5.x：认证与密码加密
    *   `winston` 3.x：日志管理
    *   `express-validator` 7.x：请求参数校验
*   **开发依赖**: ts-node-dev（热重载）、jest + ts-jest（单元测试）、supertest（API 测试）、prisma CLI（数据库迁移）
*   **代码规范**: TypeScript strict 模式；模块文件名 kebab-case；分层为 Controller → Service → Repository；DTO 用于 API 请求验证

## 4. 数据存储 (Database)
*   **主数据库**: PostgreSQL 16（Prisma 5.22 ORM，核心表包括 `Gateway`、`DeviceModel`、`PointModel`、`DeviceInstance`、`SyncRecord` 等；Schema 位于 `backend/prisma/schema.prisma`）
*   **缓存**: Redis 7（会话管理、心跳时间戳 `heartbeat:{gatewayId}`、Token 过期标记、下发锁 `sync:dispatching:{id}`）
*   **网关侧缓存**: SQLite（Node-RED 插件本地使用，用于断网场景下缓存采集数据；表结构见产品概述中的断网缓存模块）
*   **时序数据**: 平台侧时序库（用于存储采集到的点位历史数据，支持按点位 ID + 时间戳查询和去重）

## 5. 基础设施 & 工具 (Infra & Tools)
*   **消息总线**: EMQX 5（MQTT Broker，端口 1883 TCP / 8083 WebSocket / 18083 管理端口；网关数据通过标准 MQTT 3.1.1 / 5.0 协议上报）
*   **部署**: Docker Compose（一键启动 postgres、redis、emqx、nodered、backend、frontend 服务；详见项目根目录 docker-compose.yml）
*   **包管理**: npm
*   **Node-RED 交互协议**: HTTP + Basic Auth（调用 `/settings` 验证连接，调用 `/flows` 部署采集 flow）
*   **网关认证方式**: Basic Auth（用户名 + 密码，存储在网关配置中，由平台侧下发时配置）

## 6. 网关侧技术 (Gateway-Side)
*本项目特有：网关侧运行 Node-RED 执行环境，配置由平台动态下发。*

*   **运行环境**: Node-RED v3.x（含 node-red-contrib-s7、node-red-contrib-modbus、node-red-contrib-opcua、node-red-node-mqtt、node-red-node-sqlite 等协议节点）
*   **心跳流**: 由 `inject`（每 30 秒）+ `function`（组装 payload）+ `mqtt out` 组成；payload 包含 `{ gatewayId, timestamp, nodeRedVersion, ip, flowCount }`
*   **采集流**: 由平台通过 `/flows` API 动态部署，结构为 `inject` → 协议节点（S7 in / modbus 等） → `function`（值转换） → `mqtt out`（上报到平台 EMQX）
*   **缓存流**: 断网场景下通过 SQLite 节点缓存采集数据，网络恢复后按时间顺序补发
*   **插件设计原则**: 极简设计，仅负责心跳上报和执行平台下发的 flow；不再独立处理配置接收、数据采集等复杂功能
