## Node-RED 插件技术栈（极简设计）

### 设计原则
Node-RED 插件采用**极简设计**，只负责心跳上报，不做配置接收、数据采集等复杂功能。

### 核心功能
- **心跳上报**：定时发送网关状态到 EMQX
- **运行信息**：上报 Node-RED 版本、IP、流数量等

### 技术实现
```
┌─────────────────────────────────────────────────────────────┐
│                  Node-RED 基础流 (心跳)                      │
│                                                             │
│   ┌─────────┐     ┌──────────────────┐     ┌──────────┐  │
│   │ inject  │────▶│ assemble-heartbeat│───▶│ mqtt out │  │
│   │(每30秒) │     │   (组装payload)   │     │  (心跳)   │  │
│   └─────────┘     └──────────────────┘     └──────────┘  │
│                                                             │
│   Payload:                                                  │
│   { gatewayId, timestamp, nodeRedVersion, ip, flowCount }   │
└─────────────────────────────────────────────────────────────┘
```

### 配置文件说明
| 字段 | 说明 |
|------|------|
| gatewayId | 网关唯一标识 |
| timestamp | 心跳时间戳 |
| nodeRedVersion | Node-RED 版本 |
| ip | 网关 IP 地址 |
| flowCount | 当前 flow 数量 |

### 与后端的数据交互
| 方向 | 方式 | 说明 |
|------|------|------|
| 心跳上报 | MQTT → EMQX | 发送到 `gateway/{gatewayId}/heartbeat` |
| 采集配置下发 | HTTP | 后端 POST 到 `/flows`（由 Node-RED 内置节点执行采集） |

### 冗余设计（已废弃）
以下功能在后端通过 HTTP 直连方式实现，插件不需要：
- ❌ MQTT 配置订阅（后端 HTTP 直连部署）
- ❌ 数据采集节点（后端生成 S7/modbus 节点）
- ❌ 数据上报节点（MQTT 采集节点自带）

## 断网数据缓存技术实现

### 缓存架构

```
┌─────────────────────────────────────────────────────────────┐
│                    Node-RED 网关                             │
│                                                             │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐                │
│  │ inject  │───▶│  S7 in  │───▶│ mqtt out│                │
│  └─────────┘    └─────────┘    └────┬────┘                │
│                                     │                       │
│                            MQTT 发送成功?                   │
│                                │                            │
│                    ┌───────────┴───────────┐                │
│                    ▼                       ▼                │
│              ┌──────────┐           ┌──────────┐            │
│              │  发送成功  │           │  发送失败  │            │
│              └──────────┘           └────┬─────┘            │
│                                          │                   │
│                                          ▼                   │
│                                   ┌────────────┐            │
│                                   │  SQLite    │            │
│                                   │  (本地缓存) │            │
│                                   └────────────┘            │
│                                          │                   │
│                            网络恢复后读取缓存重发             │
└─────────────────────────────────────────────────────────────┘
```

### SQLite 缓存表结构

```sql
CREATE TABLE data_cache (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  device_instance_id TEXT NOT NULL,
  payload TEXT NOT NULL,        -- JSON 格式的采集数据
  timestamp INTEGER NOT NULL,   -- 数据采集时间戳
  created_at INTEGER NOT NULL,  -- 缓存写入时间戳
  retry_count INTEGER DEFAULT 0,
  status TEXT DEFAULT 'pending' -- pending, sent, failed
);
CREATE INDEX idx_status_timestamp ON data_cache(status, timestamp);
```

### Node-RED 缓存节点逻辑

```
MQTT Out 节点
    │
    ├── 发送成功 ──▶ 流程结束
    │
    └── 发送失败 ──▶ Function: cacheData
                            │
                            ▼
                      SQLite: 写入 data_cache 表
                            │
                            ▼
                      检查网络状态
                            │
                ┌───────────┴───────────┐
                ▼                       ▼
          网络恢复                  网络断开
                │                       │
                ▼                       ▼
          读取缓存              等待重试
          发送数据
          删除缓存
```

### 后端生成采集 Flow 的逻辑

```typescript
function generateFlow(gateway, instance, enableCache: boolean) {
  if (enableCache) {
    // 生成带缓存的 flow
    return [
      inject,
      s7Read,
      mqttOut,      // ← MQTT 发送
      cacheOnFail   // ← 发送失败时写入 SQLite
    ]
  } else {
    // 生成普通 flow
    return [
      inject,
      s7Read,
      mqttOut
    ]
  }
}
```

### 缓存配置继承逻辑

```typescript
function getCacheEnabled(instance, gateway): boolean {
  // 优先级：设备实例 > 网关 > 系统默认值
  if (instance.cacheEnabled !== undefined) {
    return instance.cacheEnabled
  }
  if (gateway.cacheEnabled !== undefined) {
    return gateway.cacheEnabled
  }
  return true // 系统默认值
}
```

### 缓存数据补发机制

| 阶段 | 触发条件 | 行为 |
|------|---------|------|
| 检测恢复 | MQTT 连接成功 | 立即读取缓存 |
| 按序重发 | 缓存记录存在 | 按 timestamp 顺序发送 |
| 删除确认 | 发送成功后 | 删除缓存记录 |
| 重试上限 | retry_count > 3 | 标记为 failed，停止重发 |
