# 产品概述

## 1. 项目愿景 (Vision)
*用一句话概括：我们要为谁，提供什么核心价值？*
> 为工业自动化工程师和 IoT 运维人员，提供一个统一管理分布式 Node-RED 边缘网关、模板化设备配置和实时数据采集的平台。

## 2. 核心价值 (Value Proposition)
*   **用户价值**：大幅降低多网关多设备的管理成本，网关批量激活、配置一键下发、断网数据零丢失，实时监控无需逐台登录网关操作。
*   **商业/技术价值**：平台+网关分层架构，平台侧集中管控（Node.js + MySQL），网关侧分布式执行（Node-RED），通过 SSE 实时推送、MQTT 数据上报、SQLite 本地缓存构建稳定可靠的工业数据采集链路。

## 3. 核心板块 (Core Modules)
*将系统拆解为关键的顶层模块（建议 2-4 个核心板块）。*

### 3.1 边缘网关管理模块
*   **描述**：负责网关的注册接入、状态维护、性能监控，是整个系统的运行基础。
*   **关键特性**：
    *   两种注册流程：nodered-plugin 主动注册 + 平台下发激活
    *   心跳状态监控（30秒/次，3分钟超时判定离线，SSE 实时推送）
    *   性能指标上报与历史趋势（CPU / 内存 / 磁盘，最长7天）
    *   网关分组管理与筛选

### 3.2 设备配置管理模块
*   **描述**：定义设备采集模板和实例配置，支持多种工业协议，实现模板化、可复用的设备管理。
*   **关键特性**：
    *   设备模板：版本管理、批量导入导出、模板复制
    *   点位定义：公共字段 + 协议特殊字段（Modbus / S7 / OPC UA / MQTT / TCP）
    *   设备实例：从模板创建、点位继承与覆盖、网关分配与变更
    *   支持协议：Modbus TCP/RTU、S7、OPC UA、MQTT、TCP

### 3.3 配置下发与同步模块
*   **描述**：将设备实例配置转化为 Node-RED Flow 并部署到网关，确保网关按预期执行数据采集。
*   **关键特性**：
    *   通过 Node-RED `/flows` API 动态生成并下发采集节点
    *   配置版本号比对：有变化才下发，无变化跳过
    *   批量下发：按网关分组并行，同网关内串行
    *   下发状态追踪、失败原因记录与展示

### 3.4 数据采集与缓存模块
*   **描述**：网关注册后持续采集设备数据，网络异常时本地缓存，恢复后按速率补发；平台侧通过 SSE 实时推送采集值到前端。
*   **关键特性**：
    *   采集数据 MQTT 上报，平台侧写入时序库
    *   SSE 实时推送：网关状态 / 性能 / 采集值 / 下发状态 / 缓存状态
    *   断网缓存：三级开关（设备级 > 网关级 > 系统级，默认关闭）
    *   缓存补发：独立 Topic (`device/{deviceId}/replay`)、按时间先后、速率可控（1-500条/秒）
    *   缓存去重：点位ID + 时间戳，防止重复写入

## 4. 目标用户 & 核心场景 (User & Scenes)

### 4.1 目标用户
*   **工业自动化工程师**：熟悉 Modbus / S7 等工业协议，负责设备采集方案设计与点位定义，核心诉求是模板化管理同类设备、支持协议特殊字段配置。
*   **IoT 系统运维人员**：负责网关部署、状态监控、配置下发和故障排查，核心诉求是批量管理、实时状态可见、操作结果可追踪。
*   **数据采集系统管理员**：关注数据完整性和连续性，核心诉求是断网场景下数据不丢失、补发流程可控、采集值可查历史趋势。

### 4.2 核心场景 (Key Use Cases)
*   **场景 1**：工业自动化工程师 在新工厂部署设备时，通过 [创建设备模板 → 定义点位 → 批量创建设备实例 → 分配网关 → 点击下发] 的操作流程，达成了 [数十台边缘网关在数分钟内完成采集配置] 的目标。
*   **场景 2**：IoT 系统运维人员 在日常巡检中，通过 [打开网关列表 → 查看在线状态和 CPU/内存 → 发现某网关离线 → 切换到实例详情 → 查看下发历史定位原因 → 重新下发] 的操作流程，达成了 [快速发现并修复网关采集问题] 的目标。
*   **场景 3**：数据采集系统管理员 在网络不稳定的工业环境中，通过 [在系统级开启断网缓存 → 配置补发速率 100条/秒] 的操作，达成了 [断网期间数据完整缓存、恢复后自动补发不遗漏] 的目标。

## 5. 关键业务流程 (Core Workflows)
*描述系统最核心的 1-2 条业务链路，这决定了技术架构的复杂度。*

**流程 A：网关接入与配置下发**
> 用户创建设备模板 → 定义点位 → 从模板创建设备实例 → 绑定网关 → 点击下发配置 → 平台生成 Node-RED Flow → 通过 Basic Auth 调用 `/flows` API 部署到网关 → 网关注册成功并开始采集数据 → 数据通过 MQTT 上报到平台 → 写入时序库

**流程 B：断网缓存与恢复**
> 网络正常：网关采集 → MQTT 上报 → 平台入库
> 网络断开：MQTT publish 失败 → 写入本地 SQLite 缓存（数据ID + 实例ID + 点位ID + 值 + 时间戳）
> 网络恢复：按采集时间排序 → 按系统配置速率补发 → 独立 Topic (`device/{deviceId}/replay`) + `replay:true` 标记 → 收到 PUBACK 后删除缓存记录

## 6. 验收标准 (Acceptance Criteria)
*项目第一阶段结束的具体标志是什么？（用于防止需求蔓延）*

*   **功能完备性**：6 个核心功能模块（边缘网关管理、设备模型管理、设备实例管理、配置下发与同步、实时状态推送、断网数据缓存）的核心流程全部跑通，用户可完成网关注册、模板创建、实例配置、下发采集、查看实时值和历史趋势的完整闭环；无严重 Bug（无法下发、数据丢失、SSE 无法推送等）。
*   **技术指标**：
    *   网关心跳 30 秒一次，状态变化通过 SSE 在 1 秒内推送到前端
    *   设备实例列表 / 网关列表页面加载 < 1 秒
    *   支持同时管理 50+ 网关、500+ 设备实例
    *   断网缓存数据在网络恢复后按配置速率（默认 100条/秒）补发

## 7. 约束条件 (Constraints)
*非功能性的硬性限制。*

*   网关侧必须运行 Node-RED 作为执行环境，插件方案需兼容 Node-RED v3.x 及以上版本
*   平台后端运行在 Node.js + Express，数据库使用 MySQL（含 Prisma ORM）+ Redis
*   MQTT Broker 使用 EMQX，数据通过标准 MQTT 3.1.1 / 5.0 协议上报
*   数据缓存存储在网关侧 SQLite 中，平台不直接访问缓存数据
*   前端使用 React（含 Zustand 状态管理 + SSE 订阅），运行在标准现代浏览器（Chrome / Edge / Firefox 最新两个版本）
*   所有与网关 / Node-RED 的交互使用 HTTP + Basic Auth 方式认证

## 8. 项目边界 (Out of Scope)
*明确本阶段**不做**什么，避免需求蔓延。*

*   暂不支持多租户 / 用户权限与角色管理（单用户系统）
*   暂不支持告警规则配置与告警通知（仅展示指标，不主动告警）
*   暂不支持移动端 App
*   暂不支持多语言国际化（仅中文）
*   暂不支持网关侧采集插件的动态热更新（需通过重新下发 flow 实现）
*   暂不支持缓存数据的异地备份与恢复
*   暂不支持设备模型的点位历史变更审计与版本回滚（仅提供版本号递增）
*   暂不支持历史数据的自定义报表与导出（仅提供基础时序查询和趋势图）
