# 边缘网关管理 · 需求文档

## 1. 功能概述

边缘网关管理是整个系统的基础模块，负责管理分布式部署的 Node-RED 实例（以下简称"网关"）。它承担了网关的注册与接入、实时状态监控、心跳检测、断网数据缓存、性能监控等核心能力，为上层设备模型管理、配置下发等模块提供运行基础。

## 2. 场景与用户

### 2.1 目标用户

- 工业自动化工程师 — 负责部署和调试边缘网关
- IoT 系统运维人员 — 负责日常运维和状态监控
- 数据采集系统管理员 — 负责批量管理和配置下发

### 2.2 典型场景

**场景一：新网关接入**
工程师在现场部署了一台 Node-RED 网关，在网关的 nodered-plugin 中填写平台地址和认证 token，网关自动注册到平台，平台管理员在列表中即可看到该网关，后续可修改名称、分组等。

**场景二：平台批量部署**
管理员在平台批量导入多台 Node-RED 的连接信息（地址、端口、用户名密码等），点击激活后，平台自动为每台网关部署 nodered-plugin 并推送配置，无需逐台手动配置。

**场景三：日常状态巡检**
运维人员打开网关列表，查看各网关在线状态、Node-RED 版本、流数、最后心跳时间，按分组筛选，关注离线网关并排查问题。

**场景四：断网缓存**
某现场网关与平台网络临时中断，网关仍在采集现场设备数据，这些数据在本地缓存。网络恢复后，缓存数据按时间先后顺序补发至平台。

**场景五：性能监控**
运维人员关注某网关的运行状态，进入详情页查看该网关的 CPU、内存、磁盘使用率及剩余磁盘空间的历史趋势，判断网关是否健康运行。

## 3. 核心流程

### 3.1 流程1：主动注册（主要方式）

1. 工程师在 Node-RED 上部署 nodered-plugin 插件
2. 在插件中配置平台地址、认证 token、网关名称
3. 插件生成唯一网关标识（ID），向平台发起注册请求
4. 平台按网关标识匹配：
   - 新网关 → 新增一条网关记录，状态为在线
   - 已有网关 → 更新该网关的上报信息（心跳时间、版本、IP、流数、性能指标等）
5. 网关每 30 秒上报一次心跳，平台实时更新状态

### 3.2 流程2：平台下发（次要方式）

1. 管理员在平台填写网关的 Node-RED 地址、端口、Basic Auth 用户名密码、网关名称、认证 token
2. 点击激活
3. 平台通过 HTTP 调用 Node-RED API：
   - 部署 nodered-plugin 插件
   - 推送平台地址、网关 ID、认证 token 等配置
4. 激活成功后，网关开始上报心跳，网关列表新增一行，状态为在线

### 3.3 断网缓存流程

1. **正常网络**：采集数据 → MQTT 发送 → EMQX → 平台处理
2. **网络断开**：采集数据 → MQTT 发送失败 → 写入网关本地缓存（按每条记录缓存）
3. **网络恢复**：读取缓存 → 按时间先后顺序、按系统级速率限制补发 → 删除已发送的缓存记录

### 3.4 性能监控流程

1. 网关每 30 秒上报心跳，消息中附带 CPU 使用率、内存使用率、磁盘使用率、磁盘剩余空间
2. 平台记录并存储性能历史数据
3. 用户进入网关详情页可查看各指标的历史趋势图（最长 7 天）
4. 超过 7 天保留期的性能数据自动清理

## 4. 边界与异常处理

### 4.1 注册与激活

| 异常场景 | 处理方式 |
|---------|---------|
| 流程2激活时 Node-RED 地址不可达 | 弹窗报错，提示具体失败原因 |
| 流程2激活时 Basic Auth 认证失败 | 弹窗报错，提示认证失败 |
| 流程2激活时 Node-RED API 调用失败 | 弹窗报错，提示 API 调用失败 |
| 已存在的网关再次注册 | 按网关标识匹配，更新原有记录，不新增 |

### 4.2 心跳与状态

| 异常场景 | 处理方式 |
|---------|---------|
| 超过 3 分钟无心跳 | 网关状态从在线变为离线，前端实时刷新 |
| 离线网关恢复心跳 | 网关状态从离线变回在线，前端实时刷新 |

### 4.3 断网缓存

| 异常场景 | 处理方式 |
|---------|---------|
| 缓存数据超过 15 天 | 过期数据自动清理 |
| 缓存数据达到物理存储上限 | 停止写入新数据或覆盖最旧数据，以不超过物理上限为准 |
| 用户调整缓存开关 | 新配置立即生效，无需重新下发配置 |

### 4.4 其他

| 异常场景 | 处理方式 |
|---------|---------|
| 用户删除网关 | 网关记录被移除，已关联的设备实例保留 |
| 测试连接某一项失败 | 弹窗显示失败项和错误原因，其他已测项目仍显示结果 |
| 性能数据超过 7 天 | 系统自动清理过期数据 |

## 5. 业务规则

### 5.1 状态判定规则

- 状态仅两种：在线 / 离线
- 超过 3 分钟（6 个心跳周期）未收到心跳 → 离线
- 有心跳 → 在线
- 心跳上报频率：每 30 秒一次

### 5.2 缓存开关优先级

优先级从高到低：
1. 设备级开关
2. 网关级开关
3. 系统级开关
- 三级默认均为关闭，按需开启
- 设备级有值则使用设备级，无值则使用网关级，仍无值则使用系统级

### 5.3 缓存规则

- 每条采集数据都缓存
- 保留 15 天内的数据
- 同时受物理存储上限约束，达到上限时不再写入或覆盖最旧数据
- 网络恢复后严格按时间先后顺序补发（最早的数据先补发）
- 补发速率遵守系统级配置（每秒 N 条，可配置）

### 5.4 性能数据规则

- 随心跳一同上报：CPU 使用率（%）、内存使用率（%）、磁盘使用率（%）、磁盘剩余空间
- 性能历史数据最长保留 7 天，超过自动清理
- 先不做告警，仅展示指标

## 6. 范围界定

### 6.1 本次做

- 两种注册流程（主动注册 + 平台下发激活）
- 网关列表管理：新增、删除、编辑（名称、分组、缓存开关）、分组筛选
- 测试连接：Node-RED 可达性、MQTT 连接、插件状态三项测试
- 心跳状态监控：在线 / 离线两种状态，3 分钟超时判定，SSE 实时推送
- 断网缓存：三级开关、状态展示、补发控制
- 性能监控：CPU / 内存 / 磁盘指标上报与历史趋势展示（最长 7 天）
- 网关列表信息：名称、状态、Node-RED 版本、IP、流数、最后心跳时间、分组

### 6.2 本次不做

- 设备模型管理（另一个功能模块）
- 设备实例管理（另一个功能模块）
- 配置下发与同步（另一个功能模块）
- 用户权限与角色管理
- 性能告警（仅展示指标）

## 7. 验收标准（AC）

### 7.1 Happy Path

| 编号 | 验收标准 |
|------|---------|
| AC-001 | Given 平台上还没有该网关记录，When nodered-plugin 首次注册（含网关唯一ID、名称、认证token），Then 平台新增一条网关记录，状态为在线，网关列表新增一行。 |
| AC-002 | Given 平台已有该网关记录，When nodered-plugin 再次上报（网关标识相同），Then 平台更新该网关的上报信息（心跳时间、版本、IP、流数、性能指标），不新增记录。 |
| AC-003 | Given 用户在平台填写了合法的 Node-RED 地址、端口、用户名密码、网关名称、token，When 点击激活，Then 平台成功部署 nodered-plugin 并推送配置，网关列表新增一行，状态为在线。 |
| AC-004 | Given 网关在线，When 用户点击测试连接，Then 系统依次测试 Node-RED 可达性、MQTT 连接、插件状态，弹窗显示全部测试结果。 |
| AC-005 | Given 网关当前在线，When 超过3分钟未收到新的心跳，Then 网关状态变为离线，前端列表实时刷新。 |
| AC-006 | Given 网关当前离线，When 收到该网关的心跳，Then 网关状态变回在线，前端列表实时刷新。 |
| AC-007 | Given 网关在列表中，When 用户点击编辑，Then 用户可以修改网关名称、分组、缓存开关，保存后立即生效。 |
| AC-008 | Given 断网缓存按设备开启，When 网关与平台网络断开且采集到数据，Then 数据被写入网关本地缓存。 |
| AC-009 | Given 网关本地有缓存数据，When 网络恢复，Then 系统按时间先后顺序、按可配置速率补发缓存数据，成功后清除已发送记录。 |
| AC-010 | Given 用户在网关列表查看，Then 每一行显示网关名称、状态、Node-RED 版本、IP、流数、最后心跳时间、分组。 |
| AC-021 | Given 网关在线，When nodered-plugin 上报心跳（含 CPU 使用率、内存使用率、磁盘使用率、磁盘剩余空间），Then 平台记录并存储这些性能指标。 |
| AC-022 | Given 用户在网关列表点击某网关查看详情，Then 能看到该网关的性能指标趋势图（CPU/内存/磁盘），时间范围可选择（最长7天）。 |
| AC-023 | Given 性能数据超过7天保留期，Then 系统自动清理过期数据。 |

### 7.2 Edge & Error Cases

| 编号 | 验收标准 |
|------|---------|
| AC-011 | Given 用户在平台填写的 Node-RED 地址无法连接或认证错误，When 点击激活，Then 弹窗报错，提示具体失败原因（地址不可达 / 认证失败 / API调用失败）。 |
| AC-012 | Given 测试连接时某一项失败，Then 弹窗显示失败项和错误原因，其他已测项目仍显示结果。 |
| AC-013 | Given 用户删除一个网关，Then 网关记录被移除，已关联的设备实例保留不变。 |
| AC-014 | Given 缓存数据超过15天，Then 过期数据被自动清理。 |
| AC-015 | Given 缓存数据达到物理存储上限，Then 停止写入新数据或覆盖最旧数据，以不超过物理上限为准。 |
| AC-016 | Given 用户调整系统级/网关级/设备级缓存开关，Then 新配置立即生效，无需重新下发配置。 |
| AC-017 | Given 用户在网关列表按分组筛选，Then 只显示该分组下的网关。 |

### 7.3 Business Rules

| 编号 | 验收标准 |
|------|---------|
| AC-018 | Given 任何场景，When 判定缓存是否启用，Then 遵循优先级：设备级开关 → 网关级开关 → 系统级开关（默认关）。 |
| AC-019 | Given 网关状态判定，Then 超过3分钟无心跳视为离线，有心跳视为在线，状态仅这两种。 |
| AC-020 | Given 缓存补发，Then 严格按时间先后顺序补发（最早的数据先补发），并遵守系统级速率限制。 |
| AC-024 | Given 心跳上报，Then 心跳频率固定为每30秒一次，超过3分钟（6个周期）无心跳视为离线。 |
