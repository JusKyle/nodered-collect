# 断网数据缓存 · 需求文档

## 1. 功能概述

断网数据缓存解决边缘网关与平台网络中断时，本地采集数据不丢失的问题。网关在网络断开时将采集数据缓存到本地 SQLite，网络恢复后按时间顺序、按可配置速率补发到平台。三级开关（设备级/网关级/系统级）控制缓存行为，默认关闭。平台侧展示缓存状态，支持手动清空缓存。

## 2. 场景与用户

### 2.1 目标用户

- 工业自动化工程师 — 负责配置缓存开关和补发速率
- IoT 系统运维人员 — 关注缓存状态和补发进度
- 数据采集系统管理员 — 处理异常网关的缓存数据

### 2.2 典型场景

**场景一：临时断网不影响数据完整**
某工厂网关和平台的网络临时中断 2 小时，网关仍在不停采集现场设备的点位数据。数据全部缓存在本地 SQLite 中。网络恢复后，缓存数据按时间顺序逐条补发到平台，数据不丢失。

**场景二：补发过程中再次断网**
补发进行到一半时网络再次中断，恢复后系统仅补发未发送的记录，已成功补发的记录不重复。

**场景三：某设备关闭缓存**
某一高频采集设备因数据量大且实时性要求不高，管理员在设备实例上关闭缓存开关。该设备的数据在网络断开时不缓存，其他设备正常缓存。

**场景四：缓存超限自动清理**
某网关缓存数据积压超过 15 天，系统自动清理过期数据。同时存储空间接近物理上限，新数据写入时自动覆盖最旧数据。

**场景五：运维人员查看缓存状态**
运维人员打开网关详情页，查看缓存条数、占用空间、补发进度等信息。对缓存积压的网关，手动执行清空缓存操作。

## 3. 核心流程

### 3.1 缓存触发流程

1. 网关正常运行时，采集数据通过 MQTT 发布到平台
2. 网关检测 MQTT 连接断开或 publish 失败
3. 系统检查当前设备实例的缓存开关：
   - 设备级开关有值 → 使用设备级设置
   - 无值 → 使用网关级开关
   - 仍无值 → 使用系统级开关
4. 缓存开关开启 → 该设备实例的采集数据写入本地缓存
5. 缓存开关关闭 → 数据丢弃（与网络正常时行为一致）

### 3.2 缓存存储流程

1. 网关侧使用 SQLite 作为缓存存储介质
2. 每条缓存记录包含：数据ID、设备实例ID、点位ID、值、采集时间戳、缓存时间、是否已补发标记
3. 缓存保留 15 天内的数据
4. 同时受物理存储上限约束，达到上限时停止写入或覆盖最旧数据
5. 同一设备同一点位同一时间戳的缓存记录去重，只保留一条

### 3.3 补发流程

1. 网关检测 MQTT 连接恢复
2. 系统自动启动补发流程
3. 从缓存中查询未标记为已补发的记录，按采集时间先后排序
4. 按系统级补发速率（每秒 N 条）发送到独立 Topic：`device/{deviceId}/replay`
5. 每条补发消息包含 `replay: true` 标记，与正常采集数据区分
6. 收到 MQTT PUBACK 后，将该记录标记为已补发并删除
7. 补发过程中网络再次断开，下次网络恢复后从未补发的记录继续
8. 补发时去重：相同点位ID + 相同时间戳的记录只补发一次

### 3.4 平台处理补发数据流程

1. 平台接收 Topic 为 `device/{deviceId}/replay` 的数据
2. 识别 `replay: true` 标记 → 标记为补发数据
3. 去重检查：该点位ID + 时间戳是否已写入时序库
4. 已存在 → 跳过，不重复写入
5. 不存在 → 写入时序库

### 3.5 清空缓存流程

1. 用户在平台点击"清空缓存"
2. 平台向网关发送清空指令
3. 网关清空本地 SQLite 中的所有缓存记录
4. 清空后网关继续采集，如 MQTT 未恢复则继续缓存新数据

## 4. 缓存字段定义

### 4.1 缓存记录字段

| 字段 | 类型 | 说明 |
|------|------|------|
| 数据ID | 字符串 | 缓存记录的唯一标识 |
| 设备实例ID | 字符串 | 所属设备实例 |
| 点位ID | 字符串 | 采集点位ID |
| 值 | 任意 | 采集到的点位值 |
| 采集时间戳 | 整数 | 数据采集的时间 |
| 缓存时间 | 整数 | 数据写入缓存的时间 |
| 是否已补发 | 布尔 | false=未补发，true=已补发 |

### 4.2 补发消息字段

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| deviceInstanceId | 字符串 | 是 | 设备实例ID |
| pointId | 字符串 | 是 | 点位ID |
| value | 任意 | 是 | 点位值 |
| timestamp | 整数 | 是 | 采集时间戳 |
| replay | 布尔 | 是 | 固定为 true，标记为补发数据 |

## 5. 边界与异常处理

### 5.1 缓存开关

| 异常场景 | 处理方式 |
|---------|---------|
| 三级开关均关闭 | 不缓存，数据在网络断开时丢弃 |
| 调整开关 | 立即生效，无需重新下发配置 |

### 5.2 缓存存储

| 异常场景 | 处理方式 |
|---------|---------|
| 缓存数据超过 15 天 | 过期数据自动清理 |
| 缓存达到物理存储上限 | 停止写入或覆盖最旧数据，以不超过物理上限为准 |
| 同一点位同一时间戳多条缓存 | 去重，只保留一条 |

### 5.3 补发

| 异常场景 | 处理方式 |
|---------|---------|
| 补发过程中网络再次断开 | 已补发的记录不重复发送，网络恢复后从未补发记录继续 |
| 补发数据到达后平台已有时序库记录 | 跳过，不重复写入 |
| 补发速率调整 | 立即生效，无需重启网关或重新下发配置 |

### 5.4 清空缓存

| 异常场景 | 处理方式 |
|---------|---------|
| 点击清空缓存时网关离线 | 清空指令缓存到网关，网关上线后执行清空 |
| 清空缓存后继续采集 | 新数据正常缓存（如 MQTT 仍未恢复） |

## 6. 业务规则

### 6.1 缓存开关优先级

- 设备级开关 > 网关级开关 > 系统级开关
- 有值时使用该级别设置
- 无值时向上级查询
- 三级默认均为关闭

### 6.2 缓存存储规则

- 每条采集数据都缓存（开关开启时）
- 保留 15 天内的数据
- 受物理存储上限约束
- 按点位ID + 采集时间戳去重

### 6.3 补发规则

- 严格按采集时间先后顺序补发（最早的数据先补发）
- 使用独立 Topic：`device/{deviceId}/replay`
- 消息带 `replay: true` 标记
- 收到 MQTT PUBACK 后删除缓存记录
- 补发速率：1-500 条/秒，可在系统设置中配置
- 补发速率调整立即生效

### 6.4 清空缓存规则

- 清空操作删除网关本地所有缓存记录
- 清空后不影响后续正常缓存行为
- 网关离线时清空指令缓存，上线后执行

## 7. 范围界定

### 7.1 本次做

- 网关侧缓存触发：MQTT 断开或 publish 失败
- 缓存存储：SQLite、每条记录缓存、15天保留、物理上限
- 三级缓存开关：设备级/网关级/系统级、默认关闭、立即生效
- 补发：时间顺序、独立 Topic、replay 标记、速率控制（1-500条/秒）
- 补出去重：点位ID+时间戳
- 平台处理补发数据：Topic 识别、去重检查、写入时序库
- 缓存状态展示：网关列表页（缓存条数）、网关详情页（条数+空间+补发进度）
- 清空缓存：平台手动操作

### 7.2 本次不做

- 缓存数据分析与报表
- 补发优先级策略（仅时间顺序）
- 缓存数据异地备份
- 缓存过期数据的手动恢复

## 8. 验收标准（AC）

### 8.1 Happy Path

| 编号 | 验收标准 |
|------|---------|
| AC-001 | Given 网关启用断网缓存（设备级/网关级/系统级任一开启），When 网关 MQTT 连接断开或 publish 失败，Then 采集数据自动写入网关本地缓存。 |
| AC-002 | Given 网关处于缓存模式，When 网关采集到新的点位数据，Then 数据作为一条缓存记录写入 SQLite（含数据ID、设备实例ID、点位ID、值、时间戳、缓存时间）。 |
| AC-003 | Given 网关 MQTT 连接恢复，When 本地有未补发的缓存记录，Then 网关自动启动补发流程。 |
| AC-004 | Given 补发流程已启动，When 补发数据发送，Then 每条数据发布到独立 Topic：`device/{deviceId}/replay`，消息含 `replay: true` 标记。 |
| AC-005 | Given 补发流程进行中，When 收到 MQTT PUBACK，Then 该条记录标记为已补发并从缓存中删除。 |
| AC-006 | Given 补发流程进行中，When 多个待补发记录，Then 严格按采集时间先后顺序补发（最早的数据最先补发）。 |
| AC-007 | Given 补发数据到达平台，When 该数据的时间戳在时序库中已存在记录，Then 跳过不写入。 |
| AC-008 | Given 补发数据到达平台，When 该数据的时间戳在时序库中不存在，Then 写入时序库。 |
| AC-009 | Given 用户打开网关列表，Then 每行显示该网关的当前缓存条数。 |
| AC-010 | Given 用户打开网关详情页，Then 能查看缓存条数、存储空间、补发进度。 |
| AC-011 | Given 系统中存在多条缓存记录，When 用户点击"清空缓存"，Then 网关本地所有缓存记录被清除。 |
| AC-012 | Given 补发速率设置为 100 条/秒，When 补发流程进行，Then 网关每秒最多发送 100 条缓存数据。 |

### 8.2 Edge & Error Cases

| 编号 | 验收标准 |
|------|---------|
| AC-013 | Given 补发过程中网络再次断开，When 网络恢复后继续补发，Then 已补发的记录不重复发送，从未补发的记录开始继续。 |
| AC-014 | Given 补发速率设置为 500 条/秒，When 补发流程进行，Then 网关遵守该速率限制，不超过 500 条/秒。 |
| AC-015 | Given 缓存数据超过 15 天，When 网关检查缓存，Then 过期数据自动清理。 |
| AC-016 | Given 缓存数据达到物理存储上限，When 新采集数据需要写入，Then 停止写入或覆盖最旧数据，以不超过物理上限为准。 |
| AC-017 | Given 用户调整系统级补发速率，When 下次补发执行，Then 新速率立即生效，无需重启网关或重新下发配置。 |
| AC-018 | Given 缓存中存在多条相同点位ID+相同时间戳的记录，When 补发到达该记录，Then 去重后只补发一条。 |
| AC-019 | Given 用户清空缓存后，When 网关继续采集数据且 MQTT 仍未恢复，Then 新数据继续写入缓存（清空不影响后续缓存行为）。 |
| AC-020 | Given 用户点击清空缓存，When 网关离线，Then 清空指令缓存到网关，网关上线后执行清空。 |

### 8.3 Business Rules

| 编号 | 验收标准 |
|------|---------|
| AC-021 | Given 缓存开关判定，Then 优先级：设备级 > 网关级 > 系统级，三级默认均为关闭。 |
| AC-022 | Given 缓存生命周期，Then 保留 15 天，受物理存储上限约束。 |
| AC-023 | Given 补发数据 Topic，Then 补发数据使用独立 Topic `device/{deviceId}/replay` 上报，与正常采集数据区分。 |
| AC-024 | Given 补发消息，Then 每条补发消息包含 `replay: true` 标记，平台据此判断数据来源。 |
| AC-025 | Given 补发完成条件，Then 缓存记录在收到 MQTT PUBACK 后标记为已补发并删除（非等待平台写入时序库）。 |
| AC-026 | Given 补发速率，Then 速率范围为 1-500 条/秒，在系统设置页面配置，修改后立即生效。 |
| AC-027 | Given 补出去重，Then 按点位ID + 采集时间戳去重，同一点位同一时间戳只补发一次。 |
