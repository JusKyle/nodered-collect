# 断网数据缓存 — 功能需求说明书

## 1. 背景与价值

**为什么做**：工业环境网络不稳定，断网期间数据如果不缓存将造成数据缺口，影响后续分析和决策。断网数据缓存解决边缘网关与平台网络中断时，本地采集数据不丢失的问题。

**做成什么样**：网关在网络断开时自动将数据写入本地 SQLite，网络恢复后按时间顺序、按可配置速率补发到平台，保证数据完整性。

**关联**：被边缘网关管理模块依赖。

---

## 2. 范围

### 本次做

- 断网检测：MQTT 连接断开 或 心跳发送失败（N 次重试后）
- 本地缓存：网关 SQLite 存储，字段包含设备实例 ID、点位 ID、点位值、采集时间戳、数据质量
- 两级缓存开关：网关级 > 平台级，默认均为关闭，保存期限和补发速率可配置
- 缓存补发：MQTT 重连成功后自动立即补发，按采集时间顺序，按配置速率补发
- 补发 Topic：使用独立 Topic `device/{deviceId}/replay`，消息含 `replay: true` 标记
- 补发去重：按点位 ID + 采集时间戳去重
- 平台侧展示：缓存开关状态、缓存数据条数、缓存数据总大小、补发进度、历史补发记录
- 平台侧操作：清空网关缓存（需二次确认）
- 缓存清理：过期数据自动清理、存储上限保护

### 本次不做

- 缓存数据分析与报表
- 补发优先级策略（仅时间顺序）
- 缓存数据异地备份
- 缓存过期数据的手动恢复
- 设备级缓存开关 — 本次只做两级开关

## 4. 需求详述

### 4.1 核心流程

> 这里只描述**主线路径的全局视角**——用户从哪里开始、经过哪些关键节点、到哪里结束。

**配置生效流程**
1. 管理员在「系统管理」→「网关配置」页面配置平台级默认值（缓存开关、保存期限、补发速率）
2. 管理员在「网关详情」页配置网关级覆盖值（可选）
3. 配置立即生效，无需重启网关
4. 判定优先级：网关级 > 平台级

**断网检测与缓存触发流程**
1. 网关持续监测 MQTT 连接状态和心跳发送状态
2. 当 MQTT 连接断开 或 心跳发送失败（N 次重试后），判定为断网
3. 系统检查缓存开关（网关级 > 平台级）
4. 开关开启时，将数据写入本地 SQLite
5. 存储字段：设备实例 ID、点位 ID、点位值、采集时间戳、数据质量
6. 检查存储上限，达到上限时停止写入或覆盖最旧数据

**缓存补发恢复流程**
1. 网关检测到 MQTT 重连成功
2. 立即开始补发待发送的缓存数据
3. 按采集时间先后顺序补发（最早的数据先补发）
4. 按配置速率补发（1-500 条/秒）
5. 使用独立 Topic `device/{deviceId}/replay`，消息含 `replay: true` 标记
6. 收到平台 PUBACK 确认后删除本地缓存记录
7. 补发过程中网络再断则从未补发记录继续

**清空缓存流程**
1. 用户在平台点击「清空缓存」
2. 弹出二次确认弹窗
3. 用户确认后，平台向网关发送清空指令
4. 网关清空本地所有缓存记录

### 4.2 交互规则

- 开关切换后立即生效，无需确认
- 清空缓存需二次确认弹窗
- 补发进度实时更新
- 网关离线时清空指令缓存到网关，上线后执行

### 4.3 业务规则

| 编号 | 规则 |
|------|------|
| BR-001 | 两级缓存开关优先级：网关级 > 平台级，三级默认均为关闭 |
| BR-002 | 缓存保留期限：平台级默认 15 天，可被网关覆盖 |
| BR-003 | 存储上限：达到物理存储上限时，停止写入新数据或覆盖最旧数据 |
| BR-004 | 补发触发：MQTT 重连成功后自动立即补发 |
| BR-005 | 补发顺序：严格按采集时间先后顺序（最早先补发） |
| BR-006 | 补发速率：1-500 条/秒，平台级默认 100 条/秒，可被网关覆盖 |
| BR-007 | 补发 Topic：使用独立 Topic `device/{deviceId}/replay`，消息含 `replay: true` 标记 |
| BR-008 | 补发确认：收到平台 MQTT PUBACK 后删除本地缓存记录（非等待平台写入时序库） |
| BR-009 | 去重规则：按点位 ID + 采集时间戳去重，同一点位同一时间戳只补发一次 |
| BR-010 | 配置优先级：网关级配置 > 平台级配置 |
| BR-011 | 配置生效：保存后立即生效，无需重新下发配置 |

## 5. 验收标准

> AC 是这份文档的核心交付物。后续的技术方案、任务拆分、TDD 测试用例全部从这里派生。

### 5.1 正常流程 (Happy Path)

- [ ] **AC-001**: 平台配置缓存默认值
  - **Given** 管理员在「系统管理」→「网关配置」页面
  - **When** 配置缓存开关、保存期限（天）、补发速率（条/秒），点击保存
  - **Then** 配置保存成功，立即生效

- [ ] **AC-002**: 网关覆盖平台缓存开关
  - **Given** 平台级缓存开关为关闭
  - **When** 管理员在网关详情页开启网关级缓存开关
  - **Then** 该网关使用网关级开关（开启），平台级配置不生效

- [ ] **AC-003**: 网关覆盖平台补发速率
  - **Given** 平台级补发速率为 100 条/秒
  - **When** 管理员在网关详情页配置网关级补发速率为 200 条/秒
  - **Then** 该网关按 200 条/秒补发

- [ ] **AC-004**: 断网触发缓存
  - **Given** 网关在线，MQTT 连接正常，缓存开关开启
  - **When** MQTT 连接断开（断网）
  - **Then** 网关开始将采集数据写入本地 SQLite 缓存

- [ ] **AC-005**: 断网触发缓存（心跳失败）
  - **Given** 网关在线，MQTT 连接正常，缓存开关开启
  - **When** 心跳发送失败（N 次重试后）
  - **Then** 网关开始将采集数据写入本地 SQLite 缓存

- [ ] **AC-006**: 写入缓存记录
  - **Given** 网关处于缓存模式
  - **When** 网关采集到新的点位数据
  - **Then** 数据作为一条缓存记录写入 SQLite（含设备实例ID、点位ID、点位值、时间戳、数据质量）

- [ ] **AC-007**: 断网恢复启动补发
  - **Given** 网关 MQTT 连接恢复
  - **When** 本地有未补发的缓存记录
  - **Then** 网关自动启动补发流程

- [ ] **AC-008**: 按时间顺序补发
  - **Given** 补发流程进行中
  - **When** 多个待补发记录
  - **Then** 严格按采集时间先后顺序补发（最早的数据最先补发）

- [ ] **AC-009**: 按速率补发
  - **Given** 补发速率设置为 100 条/秒
  - **When** 补发流程进行
  - **Then** 网关每秒最多发送 100 条缓存数据

- [ ] **AC-010**: 补发使用独立 Topic
  - **Given** 补发流程已启动
  - **When** 补发数据发送
  - **Then** 每条数据发布到独立 Topic：`device/{deviceId}/replay`，消息含 `replay: true` 标记

- [ ] **AC-011**: PUBACK 后删除记录
  - **Given** 补发流程进行中
  - **When** 收到 MQTT PUBACK
  - **Then** 该条记录标记为已补发并从缓存中删除

- [ ] **AC-012**: 查看网关缓存开关状态
  - **Given** 管理员在网关详情页
  - **When** 查看缓存状态
  - **Then** 显示该网关的缓存开关状态（网关级值及来源标注）

- [ ] **AC-013**: 查看缓存数据统计
  - **Given** 管理员在网关详情页
  - **When** 查看缓存状态
  - **Then** 显示当前缓存数据条数、缓存数据总大小

- [ ] **AC-014**: 查看补发进度
  - **Given** 网关正在补发缓存数据
  - **When** 管理员查看网关详情页
  - **Then** 显示补发进度（正在补发/已完成、已补发条数/总条数）

- [ ] **AC-015**: 查看历史补发记录
  - **Given** 管理员在网关详情页
  - **When** 查看历史补发记录
  - **Then** 显示历史补发的时间、条数、结果

- [ ] **AC-016**: 清空缓存
  - **Given** 网关本地有缓存数据
  - **When** 管理员点击「清空缓存」并二次确认
  - **Then** 网关本地缓存数据被清空

- [ ] **AC-017**: 清空缓存取消
  - **Given** 管理员在清空缓存的二次确认弹窗中
  - **When** 点击取消
  - **Then** 缓存数据保留

### 5.2 边界与异常 (Edge & Error Cases)

- [ ] **AC-018**: 补发中断后继续
  - **Given** 补发过程中网络再次断开
  - **When** 网络恢复后继续补发
  - **Then** 已补发的记录不重复发送，从未补发的记录开始继续

- [ ] **AC-019**: 速率上限校验
  - **Given** 补发速率设置为 500 条/秒
  - **When** 补发流程进行
  - **Then** 网关遵守该速率限制，不超过 500 条/秒

- [ ] **AC-020**: 缓存数据过期清理
  - **Given** 缓存数据超过保留期限（默认 15 天）
  - **When** 网关检查缓存
  - **Then** 过期数据自动清理

- [ ] **AC-021**: 存储上限处理（停止写入）
  - **Given** 缓存数据达到物理存储上限
  - **When** 新采集数据需要写入
  - **Then** 停止写入新数据，等待补发或过期清理

- [ ] **AC-022**: 存储上限处理（覆盖模式）
  - **Given** 缓存数据达到物理存储上限，配置为覆盖模式
  - **When** 新采集数据需要写入
  - **Then** 覆盖最旧的数据

- [ ] **AC-023**: 补发数据去重
  - **Given** 补发数据到达平台
  - **When** 该数据的时间戳在时序库中已存在记录
  - **Then** 跳过不写入

- [ ] **AC-024**: 补发数据写入
  - **Given** 补发数据到达平台
  - **When** 该数据的时间戳在时序库中不存在
  - **Then** 写入时序库

- [ ] **AC-025**: 清空后继续缓存
  - **Given** 用户清空缓存后
  - **When** 网关继续采集数据且 MQTT 仍未恢复
  - **Then** 新数据继续写入缓存（清空不影响后续缓存行为）

- [ ] **AC-026**: 网关离线时清空指令
  - **Given** 用户点击清空缓存
  - **When** 网关离线
  - **Then** 清空指令缓存到网关，网关上线后执行清空

### 5.3 业务规则验证 (Business Rules)

- [ ] **AC-027**: 缓存开关优先级 ← BR-001
  - **Given** 任意时刻
  - **When** 判定缓存是否启用
  - **Then** 遵循优先级：网关级开关 > 平台级开关（默认关闭）

- [ ] **AC-028**: 缓存生命周期 ← BR-002
  - **Given** 缓存生命周期
  - **Then** 保留期限由网关级配置覆盖，默认为 15 天，受物理存储上限约束

- [ ] **AC-029**: 补发 Topic 区分 ← BR-007
  - **Given** 补发数据 Topic
  - **Then** 补发数据使用独立 Topic `device/{deviceId}/replay` 上报，与正常采集数据区分

- [ ] **AC-030**: replay 标记 ← BR-007
  - **Given** 补发消息
  - **Then** 每条补发消息包含 `replay: true` 标记，平台据此判断数据来源

- [ ] **AC-031**: 删除条件 ← BR-008
  - **Given** 补发完成条件
  - **Then** 缓存记录在收到 MQTT PUBACK 后标记为已补发并删除（非等待平台写入时序库）

- [ ] **AC-032**: 补发速率范围 ← BR-006
  - **Given** 补发速率
  - **Then** 速率范围为 1-500 条/秒，平台级默认 100 条/秒

- [ ] **AC-033**: 去重规则 ← BR-009
  - **Given** 补出去重
  - **Then** 按点位 ID + 采集时间戳去重，同一点位同一时间戳只补发一次

- [ ] **AC-034**: 配置覆盖规则 ← BR-010
  - **Given** 网关级配置存在
  - **When** 判定实际配置值
  - **Then** 使用网关级配置，忽略平台级配置

- [ ] **AC-035**: 配置立即生效 ← BR-011
  - **Given** 管理员保存配置
  - **When** 配置变更
  - **Then** 新配置立即生效，无需重新下发配置
