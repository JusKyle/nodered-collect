## 2. 开发任务

### 基础设施

**阶段完成标准**：DeploymentRecord 表结构就绪，记录每次下发的状态、版本、结果等信息。

#### Task-01: 配置下发记录表 & 迁移 🔒

**通俗解释**：建一张配置下发记录表，每次下发都存一条记录，记录状态、版本、失败原因等。

**做什么**：
1. DeploymentRecord 表：id、gatewayId、instanceId、deployType（FULL/INCREMENTAL）、status（PENDING/RUNNING/SUCCESS/FAILED）、templateVersion、flowConfig（JSON，存下发的Flow快照）、errorMessage、startedAt、finishedAt、createdBy
2. 索引：gatewayId、instanceId、status、createdAt
3. 关联 Gateway 和 DeviceInstance
4. 状态枚举完整
5. flowConfig 存下发时的完整 Flow JSON，用于回滚和对比
6. 生成 migration 脚本

**涉及文件**：
- `backend/prisma/schema.prisma`
- `backend/prisma/migrations/`

**参考**：技术方案 第3章 → AC-001, AC-008

**依赖**：无

**预估工时**：60 分钟

**验证标准**：
- [ ] prisma migrate dev 执行成功
- [ ] DeploymentRecord 表存在
- [ ] status 是枚举（PENDING/RUNNING/SUCCESS/FAILED）
- [ ] deployType 是枚举（FULL/INCREMENTAL）
- [ ] flowConfig 是 JSON 类型
- [ ] 可以插入一条下发记录

### 切片一：Flow 生成引擎

**阶段完成标准**：根据设备模型和点位配置，能生成对应协议的 Node-RED Flow JSON。

---

#### Task-02: Modbus TCP Flow 生成 ⚠️

**通俗解释**：给一个 Modbus TCP 的设备，能生成对应的 Node-RED 采集流，包括 Modbus 节点、数据解析、MQTT 上报。

**做什么**：
1. generateModbusTcpFlow 函数
2. 输入：设备配置（连接信息、点位数组）
3. 输出：Node-RED Flow JSON 数组
4. 节点组成：modbus-client 配置节点 + modbus-read 节点 + function 节点（数据解析） + mqtt-out 节点
5. 支持多个点位分组（按寄存器类型和地址段优化）
6. 采集周期可配置（默认 5 秒）
7. 生成的 Flow ID 唯一，不重复
8. 单元测试：验证生成的 Flow 结构正确

**涉及文件**：
- `backend/src/services/flow-generator/modbus-tcp.flow.ts`
- `backend/src/types/flow.ts`

**参考**：技术方案 4.1节 → AC-002

**依赖**：Task-01

**预估工时**：90 分钟

**验证标准**：
- [ ] 输入 10 个 Modbus TCP 点位 → 输出 Flow JSON 数组
- [ ] Flow 包含 modbus-client 配置节点
- [ ] Flow 包含 modbus-read 节点，地址配置正确
- [ ] Flow 包含数据解析的 function 节点
- [ ] Flow 包含 mqtt-out 节点，主题正确
- [ ] 每个节点有唯一 ID
- [ ] 节点连线关系正确
- [ ] 点位 0 个 → 生成空 Flow（只有配置节点）

### 切片二：单设备配置下发

#### Task-09: 下发核心逻辑 🔒⚠️

**通俗解释**：给一个设备 ID，系统生成 Flow，调用 Node-RED API 部署到对应的网关，记录下发结果。

**做什么**：
1. deployConfigToDevice 核心服务函数
2. 步骤：
   a. 校验设备存在、启用
   b. 获取设备合并点位
   c. 根据协议类型生成对应 Flow
   d. 调用 Node-RED /flows API 部署 Flow
   e. 更新 DeploymentRecord 状态
   f. 成功后更新设备状态
3. 错误处理：各步骤失败记录错误信息
4. 异步执行（耗时操作，不阻塞 API 返回）
5. 下发成功后设备状态变 ONLINE 等待数据
6. Flow 名称规范：device-{instanceId}

**涉及文件**：
- `backend/src/modules/deployment/deployment.service.ts`
- `backend/src/services/flow-generator/index.ts` （统一入口）
- `backend/src/services/node-red-api.service.ts` （Node-RED API 调用）

**参考**：技术方案 5节 → AC-001, AC-003

**依赖**：Task-02 ~ Task-08

**预估工时**：120 分钟

**验证标准**：
- [ ] 调用 deployConfigToDevice(instanceId) → 创建 DeploymentRecord
- [ ] 状态从 PENDING → RUNNING → SUCCESS
- [ ] 调用 Node-RED API 部署 Flow（Mock 验证调用）
- [ ] Flow 内容正确（根据协议生成）
- [ ] 设备不存在 → 记录失败，错误信息正确
- [ ] 设备禁用 → 返回错误，不允许下发
- [ ] Node-RED API 调用失败 → 状态变 FAILED，记录错误信息
