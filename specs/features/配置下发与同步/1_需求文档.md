# 配置下发与同步 · 需求文档

## 1. 功能概述

配置下发与同步负责将平台侧的设备实例配置部署到边缘网关（Node-RED）上，并在配置变更时同步更新。该模块处理从平台到网关的单向配置推送，涵盖心跳流自动下发、采集 flow 手动下发、配置版本号比对、批量下发、删除清理等场景。采集数据通过网关的 MQTT 节点上报到平台侧 Broker，由平台写入时序库，形成完整的采集链路。

## 2. 场景与用户

### 2.1 目标用户

- 工业自动化工程师 — 负责配置下发和调试
- IoT 系统运维人员 — 负责日常下发管理
- 数据采集系统管理员 — 负责批量下发和状态追踪

### 2.2 典型场景

**场景一：网关激活**
管理员在平台手动添加一个网关，创建成功后系统自动下发心跳上报 flow。下发成功网关自动激活；下发失败则网关创建成功但状态为未激活，用户可在修复原因后重新激活。

**场景二：单个设备下发**
工程师为一个设备实例配置了通信参数和点位后，点击下发配置。系统比对发现配置有变化，为该实例生成独立 flow 并全量部署到绑定的网关 Node-RED，采集数据开始通过 MQTT 上报平台。

**场景三：批量下发**
管理员选择多个实例，系统自动按网关分组——不同网关并行下发，同一网关内的多个实例串行下发，避免 Node-RED 版本冲突。

**场景四：配置变更同步**
工程师修改了某实例的点位配置，实例配置版本号递增。用户点击下发，系统比对版本号发现变化，全量重新下发 flow。

**场景五：实例删除清理**
管理员删除设备实例，系统同时调用 Node-RED API 删除对应 flow 和平台实例记录。如果网关上 flow 已不存在，则跳过删除 flow 流程并记录。

**场景六：数据上报**
网关 Node-RED 上的采集配置运行后，采集数据通过 MQTT 发布到平台侧 Broker，平台接收数据并写入时序库。

## 3. 核心流程

### 3.1 心跳流下发流程（自动）

1. 用户在平台手动添加网关，填写 Node-RED 地址、端口、Basic Auth 认证信息
2. 网关注册成功，系统自动下发心跳上报 flow 到该网关
3. 下发成功 → 网关状态变为"已激活"
4. 下发失败 → 网关创建成功，但状态为"未激活"，记录失败原因

### 3.2 采集 flow 下发流程（手动）

1. 用户创建设备实例，绑定网关，配置通信参数和点位
2. 实例配置产生当前"配置版本号"（v1、v2...）
3. 用户点击下发配置
4. 系统比对该实例的配置版本号和已下发版本号：
   - 配置版本号 > 已下发版本号 → 执行下发
   - 配置版本号 = 已下发版本号 → 提示"无需下发"
5. 系统根据实例配置生成 Node-RED flow（含采集节点 + function 节点）
6. Flow 命名规则：`设备实例名称-设备ID`
7. 通过网关配置的 Basic Auth 调用 Node-RED `/flows` API 全量部署
8. 下发成功 → 更新已下发版本号，记录下发历史
9. 下发失败 → 记录失败原因，实例保持"待下发"状态

### 3.3 批量下发流程

1. 用户选择多个需要下发的设备实例
2. 系统将所有待下发实例按网关分组
3. 不同网关之间并行下发
4. 同一网关内的多个实例串行下发
5. 每个实例下发结果单独记录（成功/失败/跳过）
6. 某一实例失败不影响同网关后续实例的执行

### 3.4 实例删除流程（含 flow 清理）

1. 用户删除设备实例
2. 系统尝试调用 Node-RED API 删除该实例对应的 flow
3. 如果 flow 存在 → 成功删除 flow，删除平台实例记录
4. 如果 flow 不存在（已人工删除或从未下发过）→ 跳过删除 flow，删除平台实例记录，并记录"网关配置不存在，跳过删除 flow"

### 3.5 数据上报流程

1. Node-RED 上的采集节点（modbus-client / s7 in / opcua-client / tcp in）采集到原始数据
2. Function 节点按配置进行点位提取、值转换（乘系数 + 偏移量 + 精度）
3. MQTT out 节点将处理后的数据发布到平台侧 MQTT Broker
4. 平台接收数据并写入时序库

## 4. 边界与异常处理

### 4.1 网关激活

| 异常场景 | 处理方式 |
|---------|---------|
| 网关创建成功但心跳 flow 下发失败 | 网关创建成功，状态为"未激活"，记录失败原因 |
| 网关处于未激活状态 | 用户修复原因后手动重新激活（重新下发心跳 flow） |

### 4.2 配置下发

| 异常场景 | 处理方式 |
|---------|---------|
| 点击下发时网关离线 | 提示"网关离线，下发失败" |
| 点击下发时配置无变化 | 提示"无需下发"，不下发也不记录版本号变更 |
| 下发失败 | 记录失败原因，实例保持"待下发"状态，用户修复后可重新下发 |
| 网关上的 flow 被人为修改 | 下次下发时以平台配置为准，全量覆盖 |

### 4.3 实例删除

| 异常场景 | 处理方式 |
|---------|---------|
| 删除实例时网关离线 | 仍删除平台实例记录，记录"网关离线，未删除对应 flow" |
| 删除实例时 flow 在网关不存在 | 跳过删除 flow，记录"网关配置不存在，跳过删除 flow" |

## 5. 业务规则

### 5.1 版本号规则

- 每次实例配置变化自动产生新的配置版本号
- 版本号从 v1 开始，数字递增（v1 → v2 → v3...）
- 下发成功后，实例的已下发版本号更新为当前配置版本号

### 5.2 Flow 规则

- 每个设备实例生成一个独立 flow
- Flow 名称固定为：`设备实例名称-设备ID`
- Flow 部署到实例绑定的网关 Node-RED 上
- 每次下发为全量重新部署，非增量更新

### 5.3 批量下发规则

- 待下发实例按网关分组
- 不同网关之间并行下发
- 同一网关内串行下发
- 任一实例失败不影响同网关后续实例的执行

### 5.4 Node-RED 认证规则

- 使用网关配置中的 Basic Auth（用户名+密码）认证
- 平台调用 Node-RED 的 HTTP API（/flows）下发配置

### 5.5 数据上报规则

- 采集数据统一通过 MQTT 发布到平台侧 Broker
- 平台接收后写入时序库
- MQTT 的 Broker 地址、Topic 等信息由心跳流或系统配置统一设定

### 5.6 下发历史记录字段

- 实例名称
- 设备ID
- 网关名称
- 下发版本号
- 下发类型：新增 / 更新 / 删除 / 心跳流
- 下发状态：成功 / 失败 / 跳过
- 失败原因
- 下发时间
- 操作人

## 6. 范围界定

### 6.1 本次做

- 平台手动添加网关后的心跳 flow 自动下发
- 心跳流下发失败时网关状态为未激活
- 设备实例采集 flow 手动下发（单个 + 批量）
- 配置版本号与已下发版本号比对
- 全量重新部署实例 flow
- 批量下发按网关分组并行，同网关内串行
- 实例删除时调用 Node-RED API 删除对应 flow
- 删除时如果 flow 不存在则跳过并记录
- 下发历史记录
- 下发状态与失败原因展示
- 采集数据通过 MQTT 上报平台

### 6.2 本次不做

- 网关离线时排队等待下发
- 自动下发设备实例采集 flow
- 增量更新 flow
- 网关侧配置人工修改的检测与冲突合并
- 用户权限与角色管理
- 操作人精细化审计

## 7. 验收标准（AC）

### 7.1 Happy Path

| 编号 | 验收标准 |
|------|---------|
| AC-001 | Given 用户在平台手动添加网关，When 网关创建成功，Then 系统自动下发心跳上报 flow。 |
| AC-002 | Given 心跳上报 flow 下发成功，Then 网关状态变为"已激活"。 |
| AC-003 | Given 设备实例已绑定网关，When 用户点击下发配置，Then 系统为该实例生成独立 flow 并部署到绑定网关的 Node-RED。 |
| AC-004 | Given 设备实例已绑定网关，When 生成 flow，Then flow 名称为"设备实例名称-设备ID"。 |
| AC-005 | Given 实例配置版本号大于已下发版本号，When 用户点击下发配置，Then 系统全量重新下发该实例 flow。 |
| AC-006 | Given 实例配置版本号等于已下发版本号，When 用户点击下发配置，Then 提示"无需下发"。 |
| AC-007 | Given 用户选择多个实例批量下发，When 实例分布在不同网关，Then 系统按网关分组并行下发，不同网关互不阻塞。 |
| AC-008 | Given 用户选择多个实例批量下发，When 多个实例属于同一网关，Then 系统在该网关内串行下发。 |
| AC-009 | Given 下发配置成功，Then 记录下发历史，状态为成功，并更新该实例的已下发版本号。 |
| AC-010 | Given 用户删除设备实例，When 该实例 flow 存在于网关 Node-RED，Then 系统调用 Node-RED API 删除对应 flow，并删除平台实例记录。 |
| AC-011 | Given 下发历史列表，Then 每条记录显示实例名称、设备ID、网关名称、下发版本号、下发类型、状态、失败原因、下发时间、操作人。 |
| AC-012 | Given 采集数据流正常运行，When Node-RED 采集到数据，Then 数据通过 MQTT 发布到平台侧 Broker，平台接收后写入时序库。 |

### 7.2 Edge & Error Cases

| 编号 | 验收标准 |
|------|---------|
| AC-013 | Given 平台手动添加网关，When 心跳上报 flow 下发失败，Then 网关创建成功但状态为"未激活"，并记录失败原因。 |
| AC-014 | Given 用户点击下发配置，When 绑定网关离线，Then 下发失败，提示"网关离线，下发失败"。 |
| AC-015 | Given 下发失败，When 用户修复配置后，Then 用户可手动重新下发。 |
| AC-016 | Given 用户删除设备实例，When 网关上对应 flow 不存在，Then 平台删除实例记录，并记录"网关配置不存在，跳过删除 flow"。 |
| AC-017 | Given 网关上的 flow 被人工修改，When 用户重新下发配置，Then 平台以自身配置为准，全量覆盖对应实例 flow。 |
| AC-018 | Given 批量下发中某个实例失败，When 后续还有其他实例待下发，Then 同网关内后续实例继续执行，并在结果中标记失败项。 |
| AC-019 | Given 用户删除设备实例，When 网关离线，Then 仍删除平台实例记录，并记录"网关离线，未删除对应 flow"。 |

### 7.3 Business Rules

| 编号 | 验收标准 |
|------|---------|
| AC-020 | Given 设备实例配置变更，Then 实例配置版本号递增；下发成功后，已下发版本号更新为当前配置版本号。 |
| AC-021 | Given 设备实例配置下发，Then 每个实例对应一个独立 flow。 |
| AC-022 | Given Flow 命名规则，Then 实例 flow 名称固定为"设备实例名称-设备ID"。 |
| AC-023 | Given 批量下发，Then 按网关分组并行，同一网关内串行。 |
| AC-024 | Given 下发目标，Then 设备实例 flow 部署到该实例绑定的网关 Node-RED 上。 |
| AC-025 | Given Node-RED 认证，Then 使用网关配置中的 Basic Auth 认证信息调用 Node-RED API。 |
| AC-026 | Given 采集数据上报，Then 采集数据统一通过 MQTT 发布到平台侧 Broker，再写入时序库。 |
