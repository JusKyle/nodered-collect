# 技术选型方案：基于 Node-RED 的分布式数据采集系统

## 1. 项目概述

本项目是一个**分布式采集网关管理系统**，实现对多个 Node-RED 采集网关的集中统一管理。核心功能包括：边缘网关管理、设备模型管理、设备实例管理、配置下发与同步。

**关键技术约束**：
- 实时性要求：心跳检测 30s/次，列表 5s 自动刷新
- 通信模式：HTTP REST（配置下发）+ MQTT（数据上报）
- 数据模型：结构化（网关、设备模型、设备实例、点位）
- 边缘端：Node-RED 插件扩展

## 7. 核心技术决策说明

### 7.1 为什么选择 Node.js 作为后端？

1. **与 Node-RED 同源**：Node-RED 本身基于 Node.js，使用相同语言便于 API 集成和插件开发
2. **异步特性**：适合高并发 I/O 场景（心跳处理、配置下发）
3. **生态完善**：Express、Prisma、MQTT.js 等成熟库支持

### 7.2 为什么选择 PostgreSQL？

1. **数据模型固定**：网关、设备模型、设备实例、点位都是结构化数据
2. **事务支持**：配置下发需要保证数据一致性
3. **JSON 字段**：点位配置是半结构化数据，PostgreSQL 的 JSONB 类型支持高效查询

### 7.3 为什么选择 Redis？

1. **心跳状态缓存**：30s/次的心跳检测需要高性能读写
2. **实时数据暂存**：数据上报需要临时缓存
3. **会话管理**：用户登录状态管理

### 7.4 为什么选择 MQTT 作为数据上报协议？

1. **轻量级**：适合边缘设备低带宽场景
2. **发布/订阅模式**：支持一对多数据分发
3. **断线重连**：内置断线重连机制，适合不稳定网络
