# 开发规范

本文档定义 Node-RED 分布式采集网关集中管理系统项目的代码风格、命名约定、Git 工作流和 AI 协作协议，确保代码库长期可维护。

---

## 1. 语言与运行时要求

| 维度 | 规格 |
|------|------|
| **语言** | TypeScript 5.2，前后端统一 |
| **运行时** | Node.js >= 18.17 |
| **包管理器** | npm >= 9.0（前后端各自独立 package.json） |
| **严格模式** | `strict: true`，禁止 `any`（特殊情况须注释理由） |

## 4. TypeScript 特定规范

### 4.1 类型导入偏好

```typescript
// ✅ 正确写法：类型使用 import type
import type { Gateway, GatewayStatus } from './gateway.types'
import { GatewayService } from './gateway.service'

// ❌ 错误写法：类型和值混在一起
import { Gateway, GatewayService } from './gateway.service'
```

### 4.2 interface vs type 使用场景

| 场景 | 偏好 | 原因 |
|------|------|------|
| **对象形状定义** | `interface` | 可扩展、可合并 |
| **联合类型 / 交叉类型** | `type` | 更灵活 |
| **DTO / API 响应** | `interface` | 与 Prisma 生成的类型风格一致 |
| **工具类型** | `type` | 如 `type Nullable<T> = T | null` |

### 4.3 避免的写法

```typescript
// ❌ 避免：使用 any
const data: any = fetchData()

// ✅ 替代方案：使用具体类型或 unknown
const data: unknown = fetchData()
if (isGateway(data)) {
  // 类型收窄后使用
}

// ❌ 避免：非空断言 !
const name = user!.name

// ✅ 替代方案：可选链 + 默认值
const name = user?.name ?? 'Unknown'

// ❌ 避免：隐式 any 回调
arr.forEach(item => console.log(item))

// ✅ 替代方案：显式类型
arr.forEach((item: Gateway) => console.log(item))

// ❌ 避免：枚举使用 number
enum Status { Online = 0, Offline = 1 }

// ✅ 替代方案：枚举使用 string
enum Status { ONLINE = 'online', OFFLINE = 'offline' }
```

## 5. React 特定规范

### 5.1 组件设计原则

- **函数组件 + Hooks**：禁止 class 组件
- **单一职责**：每个组件只做一件事，不超过 300 行
- **状态最小化**：只在需要的地方引入 Zustand Store，避免 prop drilling
- **UI 与逻辑分离**：复杂逻辑抽到 hooks 或 utils

### 5.2 组件文件结构

```tsx
// 1. 类型导入
import type { GatewayListProps } from './gateway-list.types'

// 2. 依赖导入
import { useState, useEffect } from 'react'
import { useGatewayStore } from '@/stores/gateway.store'
import { GatewayCard } from './gateway-card'

// 3. 类型定义（如果未单独文件）
interface GatewayListProps {
  onRefresh?: () => void
}

// 4. 组件定义
export function GatewayList({ onRefresh }: GatewayListProps) {
  const { gateways, fetchGateways } = useGatewayStore()
  
  useEffect(() => {
    fetchGateways()
  }, [fetchGateways])
  
  return (
    <div className="grid grid-cols-3 gap-4">
      {gateways.map(gateway => (
        <GatewayCard key={gateway.id} gateway={gateway} />
      ))}
    </div>
  )
}
```

### 5.3 导入导出规则

- **命名导出优先**：`export function GatewayList()`
- **默认导出仅用于页面**：`export default function GatewayListPage()`
- **禁止混合导出**：一个文件要么全部命名导出，要么一个默认导出

### 5.4 导入排序

```tsx
// 按以下顺序分组，组间空一行：
// 1. React / 框架核心
import { useState, useEffect } from 'react'

// 2. 第三方库
import axios from 'axios'

// 3. 项目内部模块（按相对路径排序）
import { useGatewayStore } from '@/stores/gateway.store'
import { gatewayApi } from '@/api/gateway.api'

// 4. 类型导入（单独分组）
import type { Gateway } from '@/types/gateway.types'

// 5. 相对路径导入（同目录）
import { GatewayCard } from './gateway-card'
```

### 5.5 Hooks 使用规范

- **命名规则**：`use` + 功能名，如 `useGatewaySSE`
- **存放位置**：`frontend/src/hooks/`
- **返回值**：优先返回对象，便于解构和扩展
- **依赖数组**：必须完整声明所有依赖，禁止空数组

## 6. 样式规范

### 6.1 一样式方案

- **TailwindCSS**：唯一样式方案，禁止内联 style（除非动态值）
- **动态类名**：使用 `clsx` 或条件表达式，禁止字符串拼接
- **响应式**：使用 Tailwind 的 `sm/md/lg/xl` 断点

```tsx
// ✅ 正确用法
<div className={clsx('px-4', isOnline ? 'bg-green-100' : 'bg-red-100')}>
  {name}
</div>

// ❌ 禁止：内联 style
<div style={{ padding: '16px', backgroundColor: isOnline ? 'green' : 'red' }}>
```

### 6.2 禁止的样式写法

```tsx
// ❌ 禁止：内联 style（非动态值）
<div style={{ padding: '16px' }}>

// ❌ 禁止：CSS 文件（除非 Tailwind 无法覆盖）
import './gateway-list.css'

// ❌ 禁止：styled-components
const StyledDiv = styled.div`padding: 16px;`

// ✅ 正确：Tailwind 工具类
<div className="p-4">
```

## 7. Git 工作流

### 7.1 分支策略

> 简化 Trunk-Based（独立开发者模式）

```
main ──────────────────────────────────────▶
       │
       └── feat/gateway-management ──▶ (squash merge)
       │
       └── fix/mqtt-topic-matching ──▶ (squash merge)
```

| 规则 | 说明 |
|------|------|
| **主分支** | `main`，始终可部署 |
| **功能分支** | 从 `main` 拉出，完成后 squash merge 回 `main` |
| **分支生命周期** | 1-3 天，尽快合并 |
| **合并方式** | Squash Merge（保持 main 历史整洁） |

### 7.2 分支命名规范

```
<type>/<short-description>
```

| 类型 | 用途 | 示例 |
|------|------|------|
| `feat/` | 新功能 | `feat/gateway-management` |
| `fix/` | Bug 修复 | `fix/mqtt-topic-matching` |
| `refactor/` | 重构 | `refactor/sse-service` |
| `docs/` | 文档更新 | `docs/api-spec` |
| `chore/` | 构建/配置 | `chore/upgrade-deps` |

### 7.3 提交信息规范 (Conventional Commits)

所有 Git 提交信息**必须**遵循 [Conventional Commits](https://www.conventionalcommits.org/) 格式：

```
<type>(<scope>): <subject>

[body]

[footer]
```

#### 类型 (type)

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat(gateway): 添加测试连接功能` |
| `fix` | Bug 修复 | `fix(mqtt): 修复 topic 通配符匹配` |
| `refactor` | 重构（不改变功能） | `refactor(sse): 重构推送服务` |
| `style` | 代码格式调整 | `style: 统一缩进为 2 空格` |
| `docs` | 文档变更 | `docs: 更新 API 设计规范` |
| `test` | 测试相关 | `test(gateway): 添加心跳检测单元测试` |
| `chore` | 构建/工具/依赖 | `chore: 升级 TypeScript 到 5.2` |
| `perf` | 性能优化 | `perf: 优化网关列表查询` |

#### 作用域 (scope)

| scope | 对应范围 |
|-------|---------|
| `gateway` | 网关管理模块 |
| `device-model` | 设备模板模块 |
| `device-instance` | 设备实例模块 |
| `sync` | 配置下发模块 |
| `sse` | SSE 推送服务 |
| `mqtt` | MQTT 服务 |
| `heartbeat` | 心跳检测服务 |
| `cache` | 断网缓存模块 |
| `frontend` | 前端通用 |
| `backend` | 后端通用 |
| `docs` | 文档 |
| `deploy` | 部署配置 |

#### 提交信息规则

- `subject`：中文、首字母大写、无标点结尾、不超过 50 字
- `body`：解释**为什么**，不是**做了什么**（可选）
- `footer`：Issue 引用格式 `Closes #123`（可选）

#### 示例

```
feat(gateway): 添加测试连接功能

- 调用 Node-RED /settings API 验证连接
- 成功时获取 Node-RED 版本并更新状态
- 失败时返回具体错误信息

Closes #45
```

```
fix(mqtt): 修复 topic 通配符匹配逻辑

根因：topicMatches 函数参数顺序错误，receivedTopic 在前导致 '+' 通配符无法匹配
修复：调整为 subscribedTopic 在前，按 MQTT 规范逐段比对
```

## 8. 代码注释规范

### 8.1 何时写注释

| 需要注释 | 不需要注释 |
|----------|-----------|
| 业务规则解释（如心跳判定逻辑） | 代码本身自解释 |
| 非显而易见的算法 | 变量名已说明用途 |
| 第三方库的特殊用法 | 标准的 React Hooks 用法 |
| 临时方案 / TODO | 简单的 CRUD 操作 |

### 8.2 注释风格

```typescript
// ✅ 解释 "为什么"
// 心跳超时判定：3 分钟 = 6 个心跳周期，避免单次丢包误判
const OFFLINE_THRESHOLD = 6 * HEARTBEAT_INTERVAL

// ❌ 描述 "做了什么"
// 设置阈值为 180 秒
const OFFLINE_THRESHOLD = 180
```

### 8.3 TODO / FIXME / HACK 标记

```typescript
// TODO: 后续支持批量下发优化（当前串行）
// FIXME: 网关离线时下发队列会丢失，需持久化
// HACK: 临时方案：直接调用 Node-RED API，后续改为通过插件订阅
```

## 9. 错误处理规范

### 9.1 服务端错误处理

```typescript
// 统一的错误返回类型
interface ApiError {
  code: string        // 错误码，如 'VALIDATION_ERROR'
  message: string     // 用户友好消息
  details?: unknown   // 详细信息（仅开发环境）
}

// Controller 层统一处理
try {
  const result = await service.getGatewayById(id)
  return res.json({ success: true, data: result })
} catch (error) {
  if (error instanceof ValidationError) {
    return res.status(400).json({
      success: false,
      error: { code: 'VALIDATION_ERROR', message: error.message }
    })
  }
  // 未知错误，记录日志但不暴露细节
  logger.error(error)
  return res.status(500).json({
    success: false,
    error: { code: 'INTERNAL_ERROR', message: '服务器内部错误' }
  })
}
```

### 9.2 客户端错误处理

- **API 调用**：统一用 `try-catch`，失败时 toast 提示
- **表单校验**：客户端即时反馈 + 服务端二次校验
- **SSE 断开**：自动重连，10 次失败后提示刷新

```typescript
// API 调用错误处理
try {
  await gatewayApi.deleteGateway(id)
  toast.success('删除成功')
} catch (error) {
  toast.error(error.message || '删除失败')
}
```

### 9.3 安全原则

- 永远不在 UI 中暴露原始错误堆栈或数据库错误信息
- 表单校验双重执行：客户端（即时反馈）+ 服务端（安全保障）

## 13. AI 协作协议

> 本节定义 AI 辅助开发时必须遵守的规则，确保 AI 生成的代码与人工代码保持一致性。

### 13.1 AI 写代码前必读

| 文档 | 目的 |
|------|------|
| `specs/1_产品概述.md` | 理解业务背景和核心价值 |
| `specs/2_技术栈.md` | 确认语言、框架、工具链 |
| `specs/3_项目结构.md` | 理解模块划分和文件放置规则 |
| `specs/4_开发规范.md` | 本文档，遵循命名、风格、注释规范 |
| `specs/features/[对应功能]/1_需求文档.md` | 理解功能需求和验收标准 |
| 相关模块的现有代码 | 理解当前实现，保持一致性 |

### 13.2 AI 写代码的规则

| 规则 | 说明 |
|------|------|
| **读后写** | 修改文件前必须先读取该文件 |
| **最小改动** | 只修改任务要求的部分，不做"顺便优化" |
| **不猜版本** | 安装依赖时使用 `npm install <package>`，不手写版本号 |
| **不引入新依赖** | 除非任务明确要求 |
| **不写无意义注释** | 不写复述代码的注释 |
| **不创建多余文件** | 不主动创建文档或配置文件 |
| **遵循现有模式** | 相同类型的代码必须遵循已有文件的写法 |
| **提交信息规范** | 严格遵循 Conventional Commits |

### 13.3 AI 写完代码后必做

| 步骤 | 操作 |
|------|------|
| **1. 类型检查** | `npm run build` 确认无编译错误 |
| **2. Lint 检查** | `npm run lint` 确认无 Lint 错误 |
| **3. 依赖方向** | 确认没有违反目录结构中的依赖方向原则 |
| **4. 环境变量** | 如有新增，同步更新 `.env.example` |

### 13.4 错题本机制

当 AI 生成的代码出现问题时，在 `specs/错题本.md` 中记录：

| 记录项 | 说明 |
|--------|------|
| **错误类型** | 类型错误 / 逻辑错误 / 规范违规 / 依赖错误 |
| **错误描述** | 具体出了什么问题 |
| **根因分析** | 为什么会犯这个错 |
| **正确做法** | 下次应该怎么做 |

```
frontend/src/
  ├── api/ ────────────────────────▶ stores/ (调用 API 更新状态)
  ├── components/ ─────────────────▶ stores/ (读取状态)
  ├── pages/ ─────────────────────▶ components/ + api/ + hooks/ + stores/
  ├── hooks/ ─────────────────────▶ api/ + stores/
  └── stores/ ─────────────────────▶ api/ (调用 API)

backend/src/
  ├── modules/ ───────────────────▶ services/ + repositories/
  ├── services/ ──────────────────▶ repositories/ + config/
  ├── middlewares/ ───────────────▶ services/ (认证中间件)
  ├── routes/ ────────────────────▶ modules/ (路由聚合)
  └── config/ ────────────────────▶ 无依赖（底层配置）

禁止方向：
  frontend/stores/ ✗→ frontend/pages/ (状态管理不依赖页面)
  backend/services/ ✗→ backend/routes/ (服务不依赖路由)
  backend/modules/ ✗→ backend/routes/ (模块不依赖路由)
```

> 详细规则见 `specs/3_项目结构.md`

## 15. 安全编码规范

| 类别 | 规则 |
|------|------|
| **输入校验** | 所有 API 请求参数必须用 `zod` 校验，禁止信任前端数据 |
| **注入防护** | 使用 Prisma ORM，禁止拼接 SQL；MQTT topic 用白名单校验 |
| **XSS** | 前端禁止 `dangerouslySetInnerHTML`，后端返回数据自动 JSON 序列化 |
| **密钥管理** | 所有密钥通过环境变量配置，禁止硬编码 |
| **文件上传** | 当前阶段不支持文件上传 |
| **认证检查** | 所有 API（除 `/api/events` SSE）必须校验 JWT Token |
| **日志安全** | 禁止在日志中记录 Token、密码、密钥等敏感信息 |
| **错误暴露** | 禁止向前端暴露原始错误堆栈或数据库错误信息 |
