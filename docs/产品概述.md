# 产品概述：基于 Node-RED 的分布式数据采集系统

## 1. 愿景与核心价值

### 1.1 产品愿景
打造一个**分布式采集网关管理系统**，实现对多个 Node-RED 采集网关的集中统一管理。通过设备模板化和配置批量下发，大幅降低多网关场景下的配置管理成本，提升设备接入效率。

### 1.2 核心价值

| 核心价值 | 说明 |
|---------|------|
| **降低管理成本** | 多网关集中管理，统一配置入口，无需逐个登录 Node-RED |
| **模板化配置** | 设备模型模板化，一次定义多处复用，减少重复配置 |
| **混合部署** | 支持云端统一下发配置，也支持本地节点灵活微调 |
| **快速接入** | 新设备通过模板快速接入，批量下发即刻生效 |

## 2. 用户与角色

### 2.1 目标用户

| 用户角色 | 职责 | 核心诉求 |
|---------|------|---------|
| 运维人员 | 设备接入调试、网关运维、配置管理 | 高效配置、快速部署、问题定位 |
| 设备管理人员 | 设备台账管理、点位维护、配置审核 | 统一视图、批量操作、配置追溯 |

### 2.2 用户场景

**场景一：批量设备快速接入**
> 运维人员新到一批 50 台 PLC 设备，需要紧急接入采集。
> - 创建设备模板（定义点位模型）
> - 批量导入设备实例（绑定模板）
> - 一键同步到多个网关
> - 50 台设备同时开始采集

**场景二：网关配置统一管理**
> 运维人员管理 10 个边缘网关，每个网关部署在不同厂区。
> - 在「边缘网关」页面查看所有网关状态
> - 远程下发设备配置到指定网关
> - 在 Node-RED 中查看对应设备节点
> - 本地调试时可选择性微调节点

## 3. 核心板块

### 3.1 产品信息架构

```
┌─────────────────────────────────────────────────────────────┐
│                    采集系统管理平台                           │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐        │
│  │  边缘网关     │  │  设备模型     │  │  设备实例     │        │
│  │  管理        │  │  管理        │  │  管理        │        │
│  └─────────────┘  └─────────────┘  └─────────────┘        │
│         │               │               │                    │
│         ▼               ▼               ▼                    │
│  ┌─────────────────────────────────────────────────┐        │
│  │              配置下发与同步                        │        │
│  └─────────────────────────────────────────────────┘        │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 四大核心模块

#### 模块一：边缘网关管理
- 管理多个 Node-RED 实例（采集网关）
- 网关注册、状态监控（在线/离线）、心跳检测
- 在 Node-RED 配置页面展示已下发设备节点

#### 模块二：设备模型管理
- 定义设备类型模板（厂商、型号、协议）
- 定义采集点位模型（点位名称、地址、数据类型、采集频率）
- 支持点位继承与覆盖

#### 模块三：设备实例管理
- 从设备模型创建设备实例
- 继承设备模型的点位
- 支持实例级自定义点位（添加/覆盖）
- 设备实例状态管理（待同步/已下发/运行中/离线）

#### 模块四：配置下发与同步
- 将设备实例下发到指定网关
- Node-RED 侧自动生成设备节点
- 支持从 Node-RED 侧手动选择已存在的设备实例
- 配置变更同步与版本管理

## 4. 采集协议与点位模型

### 4.1 协议分层设计

| 层级 | 协议 | 说明 |
|-----|------|-----|
| **工业常用** | Modbus TCP、S7 (西门子)、OPC UA | 主流 PLC 通信 |
| **物联网** | MQTT、HTTP REST、CoAP | 云边协同 |
| **专用** | MTConnect、FOCAS (发那科) | 数控机床 |
| **扩展预留** | 自定义 TCP/UDP | 按需扩展 |

### 4.2 点位模型设计

```
采集点位
├── 基本信息
│   ├── 点位名称 (pointName)
│   ├── 点位编码 (pointCode)        -- 唯一标识
│   ├── 数据类型 (dataType: int/float/string/bool)
│   └── 读写权限 (rw: read/write/readWrite)
├── 地址定义
│   ├── 协议类型
│   ├── 寄存器地址 / 节点ID / 路径
│   └── 数据偏移 / 长度
├── 采集配置
│   ├── 采集频率 (ms，最小 100ms)
│   └── 超时时间 (ms)
└── 数据处理
    ├── 单位 (unit)
    ├── 转换公式 (formula)          -- 可选
    └── 告警阈值 (threshold)         -- 可选
```

## 5. 配置下发方案

### 5.1 下发整体架构

```
┌──────────────┐     HTTP/MQTT      ┌──────────────┐
│   管理平台    │ ──────────────────→│  Node-RED    │
│  (云端)       │   下发配置         │  网关(边缘)   │
│              │ ←──────────────────│              │
│              │   状态上报         │  ┌────────┐  │
└──────────────┘                    └──│配置节点 │──┼──→ 设备
                                       │(本系统) │  │
                                       └────────┘  │
                                       ┌────────┐  │
                                       │设备节点 │  │
                                       │(动态)  │──┼──→ 设备
                                       └────────┘  │
                                       ┌────────┐  │
                                       │数据节点 │  │
                                       └────────┘  │
```

### 5.2 配置下发流程

```
1. 管理平台 → 生成配置
   设备实例配置 → JSON格式
   {
     "deviceId": "dev_001",
     "deviceName": "PLC_1号",
     "protocol": "modbus-tcp",
     "connection": { "host": "192.168.1.10", "port": 502 },
     "points": [
       { "code": "temp", "address": "40001", "type": "float" },
       { "code": "press", "address": "40003", "type": "float" }
     ]
   }

2. 管理平台 → 调用 Node-RED API
   POST http://gateway:1880/admin/flows
   或写入指定目录的 config.json

3. Node-RED 接收配置 → 动态生成节点
   - 创建设备实例节点（配置节点）
   - 创建采集节点（协议节点）
   - 创建数据处理节点

4. 节点部署 → 开始采集
```

### 5.3 关键技术点

| 问题 | 方案 |
|-----|-----|
| 管理平台如何识别 Node-RED | 每个网关注册时存储 API URL + 认证token |
| 配置如何生效 | 调用 Node-RED `/flows` API 部署 |
| 节点如何动态生成 | 通过 `node-red` 的 REST API 动态写入 flow |

## 6. Node-RED 设备节点设计

### 6.1 节点架构

```
┌─────────────────────────────────────────────────────┐
│                   Node-RED Flow                     │
├─────────────────────────────────────────────────────┤
│                                                     │
│  ┌──────────────┐    ┌──────────────┐              │
│  │ 配置节点      │───→│ 设备实例节点  │──┐           │
│  │ (Device Mgr) │    │ (继承配置)    │  │           │
│  └──────────────┘    └──────────────┘  │           │
│         │                  │           ▼           │
│         │           ┌──────────────┐  ┌──────────┐ │
│         │           │ 协议采集节点  │→│ 数据处理 │ │
│         │           │(Modbus/S7..) │  │  节点    │ │
│         │           └──────────────┘  └──────────┘ │
│         │                                     │    │
│         │                              ┌──────┴─┐ │
│         │                              │ MQTT   │ │
│         │                              │ Output │ │
│         │                              └────────┘ │
└─────────┴─────────────────────────────────────────┘
```

### 6.2 节点职责

| 节点 | 作用 |
|-----|-----|
| **配置节点** | 存储设备实例元数据，运行时提供配置信息 |
| **设备实例节点** | 接收配置，实例化点位，管理采集生命周期 |
| **协议采集节点** | 按协议采集数据（复用现有节点如 node-red-contrib-modbus） |
| **数据处理节点** | 单位转换、阈值检测、数据格式化 |

### 6.3 手动选择设备实例

在 Node-RED 中手动添加节点时：
- 选择「设备实例」类型
- 下拉列表展示管理平台已下发的设备实例
- 选择后自动带入点位配置
- 支持后期绑定/解绑

## 7. 网关断网处理

### 7.1 断网处理流程

```
┌─────────────────────────────────────────────────────────┐
│                    断网处理流程                           │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  断网检测                                                │
│    │                                                    │
│    ▼                                                    │
│  ┌──────────────┐                                       │
│  │ 心跳超时     │──Yes──→ 标记「网关离线」                │
│  │ (30s间隔)   │           │                           │
│  └──────────────┘           │                           │
│         │No                 ▼                           │
│         │            ┌──────────────┐                   │
│         └──────────→│ 继续本地采集  │                   │
│                      │ (独立运行)    │                   │
│                      └──────────────┘                   │
│                             │                           │
│                             ▼                           │
│                      ┌──────────────┐                   │
│                      │ 数据本地缓存  │                   │
│                      │ (SQLite/文件) │                   │
│                      └──────────────┘                   │
│                             │                           │
│                             ▼                           │
│                      网络恢复                            │
│                             │                           │
│                             ▼                           │
│                      ┌──────────────┐                   │
│                      │ 缓存数据补发  │                   │
│                      │ (带时间戳)    │                   │
│                      └──────────────┘                   │
└─────────────────────────────────────────────────────────┘
```

### 7.2 具体措施

| 场景 | 处理方案 |
|-----|---------|
| **检测断网** | Node-RED 每 30s 向管理平台发心跳，3次超时判定离线 |
| **本地采集** | 断网后 Node-RED 继续独立采集，不受影响 |
| **数据缓存** | 采集数据写入本地 SQLite，断网期间数据不丢失 |
| **数据补发** | 网络恢复后，按时间顺序补发缓存数据，带原始时间戳 |
| **状态同步** | 恢复后自动同步设备状态，界面更新「网关在线」 |
| **告警通知** | 断网时管理平台产生「网关离线」告警（后期扩展） |

### 7.3 网关状态显示

```
网关状态：
  ● 在线 (绿色)
  ◯ 离线 (灰色) - 最后在线: 2026-06-16 10:30:15
```

## 9. 验收标准

### 9.1 网关管理

**AC-001**: Given 管理员在「边缘网关」页面，When 添加一个新的 Node-RED 实例（填写名称、地址、端口），Then 网关创建成功，状态显示为"在线"或"离线"。

**AC-002**: Given 网关已注册，When 网关心跳超时，Then 状态自动变为"离线"，并显示最后在线时间。

### 9.2 设备模型

**AC-003**: Given 运维人员创建设备模型，When 定义基本信息（名称、厂商、型号、协议）和点位模型，Then 模型保存成功，可在列表中查看。

**AC-004**: Given 设备模型已存在，When 在点位模型中添加测点（名称、地址、数据类型、采集频率），Then 点表保存成功，支持秒级频率。

### 9.3 设备实例

**AC-005**: Given 设备模型已创建，When 从模型创建设备实例并绑定网关，Then 实例创建成功，状态为"待同步"。

**AC-006**: Given 设备实例继承模型点位，When 在实例中自定义点位（添加/覆盖），Then 实例点位为模型点位与自定义点位的合并结果。

### 9.4 配置下发

**AC-007**: Given 设备实例已创建，When 点击「下发到网关」并选择目标网关，Then 配置同步成功，Node-RED 中自动生成对应设备节点，实例状态变为"运行中"。

**AC-008**: Given 设备实例已下发，When 在 Node-RED 中查看，Then 可看到配置好的设备节点，支持手动选择绑定设备实例。

### 9.5 批量操作

**AC-009**: Given 运维人员选择多个设备实例，When 点击「批量下发」，Then 配置同时下发到各自绑定的网关，所有设备同时开始采集。

### 9.6 断网处理

**AC-010**: Given 网关正常运行，When 网络中断，Then 管理平台显示网关「离线」，Node-RED 继续本地采集，数据缓存本地。

**AC-011**: Given 网关离线且有缓存数据，When 网络恢复，Then 缓存数据自动补发到管理平台，带原始时间戳。
