# Node-RED 边缘网关数据采集平台-FRD

Node-RED 边缘网关数据采集平台用于统一管理多台 `Node-RED` 边缘网关、设备采集模板、设备实例配置、配置下发、实时状态推送和断网数据缓存，帮助工业现场完成设备接入、采集配置部署、运行状态监控和数据连续性保障。

## 通用交互与约束

- 平台采用平台侧集中管控与网关侧分布式执行架构。
- 网关侧必须运行 `Node-RED`，插件方案需兼容 `Node-RED v3.x` 及以上版本。
- 平台后端使用 `Node.js + Express`。
- 数据库使用 `MySQL`、`Prisma ORM` 和 `Redis`。
- `MQTT Broker` 使用 `EMQX`，支持 `MQTT 3.1.1 / 5.0`。
- 网关侧断网缓存使用 `SQLite`，平台不直接访问缓存数据库。
- 前端使用 `React`、`Zustand` 和 `SSE` 订阅机制。
- 前端运行环境为 `Chrome`、`Edge`、`Firefox` 最新两个版本。
- 所有平台与网关 / `Node-RED` 的交互使用 `HTTP + Basic Auth` 认证。
- 网关心跳每 `30 秒` 上报一次，超过 `3 分钟` 未上报时判定离线。
- 网关状态变化通过 `SSE` 在 `1 秒` 内推送到前端。
- 设备实例列表和网关列表页面加载时间小于 `1 秒`。
- 平台支持同时管理 `50+ 网关` 和 `500+ 设备实例`。
- 断网缓存数据在网络恢复后按配置速率补发，默认速率为 `100 条/秒`。
- 系统不存在无法下发配置、采集数据丢失、`SSE` 无法推送等严重问题。

## 【场景故事】产品页面功能描述及交互描述

### 新工厂批量部署采集配置

- 使用角色：`工业自动化工程师`。
- 使用场景：在新工厂上线阶段，需要为多类工业设备配置采集点位并部署到数十台边缘网关。
- 操作路径：先在平台中创建设备模板，按协议定义点位字段，再基于模板批量创建设备实例并分配到对应网关。
- 下发行为：配置完成后，点击下发，平台将设备实例配置转换为 `Node-RED Flow`，并通过网关的 `/flows` API 部署到 `Node-RED` 环境。
- 业务目标：使多台网关在数分钟内完成采集配置。

### 日常巡检与故障排查

- 使用角色：`IoT 系统运维人员`。
- 使用场景：日常巡检时打开网关列表，查看所有网关的在线状态、分组、`CPU`、`内存` 和 `磁盘` 指标。
- 异常处理：当发现某台网关离线时，进入相关设备实例或下发记录页面，查看最近一次配置下发状态和失败原因。
- 恢复操作：在问题修复后重新触发下发，快速恢复采集任务。

### 网络异常下的数据连续性保障

- 使用角色：`数据采集系统管理员`。
- 使用场景：在网络不稳定的工业现场开启断网缓存能力，并配置缓存补发速率。
- 正常链路：网络正常时，网关采集数据通过 `MQTT` 上报平台。
- 断网链路：网络断开时，`MQTT publish` 失败的数据写入网关本地 `SQLite`。
- 恢复链路：网络恢复后，网关按采集时间排序，通过独立 `Topic` 补发缓存数据，并在收到确认后删除缓存记录。
- 业务目标：避免数据丢失和重复写入。

## 【整体结构和业务流程】产品页面功能描述及交互描述

### 功能结构图

- 平台总主题：`Node-RED 边缘网关数据采集平台`。
- 一级模块：`边缘网关管理`、`设备配置管理`、`配置下发与同步`、`数据采集与缓存`。
- `边缘网关管理` 包含：`网关注册接入`、`心跳状态监控`、`性能指标上报`、`网关分组管理`。
- `设备配置管理` 包含：`设备模板管理`、`点位定义`、`设备实例管理`、`协议字段配置`。
- `配置下发与同步` 包含：`配置版本比对`、`Node-RED Flow 生成`、`批量下发`、`下发状态追踪`。
- `数据采集与缓存` 包含：`MQTT 数据上报`、`SSE 实时推送`、`SQLite 本地缓存`、`缓存补发与去重`。

### 网关接入与配置下发流程

- 参与角色：`用户`、`平台`、`边缘网关`、`Node-RED`、`MQTT Broker`、`MySQL/时序库`。
- 第一步：用户在平台中创建设备模板与点位。
- 第二步：用户在平台中创建设备实例并绑定网关。
- 第三步：用户点击下发配置。
- 第四步：平台生成 `Node-RED Flow`。
- 第五步：平台通过 `Basic Auth` 调用网关 `/flows` API。
- 第六步：边缘网关将采集 `Flow` 部署到 `Node-RED`。
- 第七步：`Node-RED` 向 `MQTT Broker` 上报采集数据。
- 第八步：`MQTT Broker` 向平台推送 `MQTT` 消息。
- 第九步：平台将采集数据写入 `MySQL/时序库`。
- 第十步：平台通过 `SSE` 向用户推送状态与实时值。

### 断网缓存与恢复流程

- 第一步：网关采集设备数据。
- 第二步：判断 `MQTT` 是否可用。
- `MQTT` 可用时：发布到平台，平台写入时序库，并通过 `SSE` 推送实时值。
- `MQTT` 不可用时：写入本地 `SQLite` 缓存。
- 网络恢复后：按采集时间排序。
- 补发行为：按配置速率补发，发布到 `device/deviceId/replay`。
- 确认行为：收到 `PUBACK` 后删除缓存记录。
- 未收到 `PUBACK` 时：返回等待网络恢复流程。

## 【配置下发与同步】产品页面功能描述及交互描述

### 说明

- 页面类型：`后台管理类（Admin / Table-First）+ 分步流程/向导类（Step Wizard）`。
- 页面综述：用于将设备实例配置转换为 `Node-RED Flow`，并部署到绑定网关。

### 下发入口

- 入口位置：用户可在设备实例列表、网关详情或下发管理页触发 `下发配置`。

### 下发流程

- 第一步：选择设备实例。
- 第二步：按网关分组。
- 第三步：校验配置版本。
- 第四步：生成 `Flow`。
- 第五步：调用 `/flows` API。
- 第六步：记录下发结果。

### 批量策略

- 按网关分组并行执行。
- 同一网关内串行下发，避免 `Node-RED Flow` 被并发覆盖。

### 状态追踪

- 状态值：`待下发`、`下发中`、`成功`、`失败`、`跳过`。
- 失败记录：记录失败原因。

### 版本比对

- 当平台配置版本与网关已部署版本一致时跳过下发，减少无效部署。

## 【断网数据缓存】产品页面功能描述及交互描述

### 说明

- 页面类型：`后台管理类（Admin / Table-First）+ 复杂 Web / 工作台类（Dashboard / Workbench）`。
- 页面综述：用于配置断网缓存开关、补发速率，并查看缓存状态和补发进度。

### 配置层级

- 支持系统级、网关级、设备级三级开关。
- 优先级为设备级 > 网关级 > 系统级。
- 默认状态为关闭。

### 缓存策略

- 触发条件：`MQTT publish` 失败时。
- 缓存位置：网关本地 `SQLite`。
- 缓存字段：`数据ID`、`实例ID`、`点位ID`、`值`、`时间戳`。

### 补发策略

- 网络恢复后按采集时间先后补发。
- 补发速率支持 `1-500 条/秒`。
- 默认补发速率为 `100 条/秒`。

### 去重策略

- 平台侧按 `点位ID + 时间戳` 去重，避免缓存补发造成重复写入。

### 状态展示

- 展示字段：`缓存条数`、`最早缓存时间`、`补发速率`、`补发进度`、`最近补发结果`。

## 【产品设计风格】产品页面功能描述及交互描述

### 设计定位

- 产品采用专业、稳定、信息密度适中的工业物联网后台风格。
- 产品优先保证状态可读性、异常可发现性和批量操作效率。
- 界面不追求强视觉装饰，应突出实时状态、配置关系、下发结果和数据趋势。

### 布局原则

- 列表页采用表格优先布局，筛选区置顶，批量操作靠近列表工具栏。
- 详情页采用概览卡片 + 分区信息布局，优先展示在线状态、最近心跳、绑定实例、下发状态和关键指标。
- 工作台和监控视图采用卡片化布局，分别承载网关状态、性能趋势、实时采集值和缓存状态。
- 高风险操作如重新下发、重新激活、覆盖配置需要明确二次确认和结果反馈。

### 视觉建议

- 主色建议使用工业科技感较强的蓝色系，用于主按钮、链接、选中状态和关键指标强调。
- 成功状态使用绿色，表示在线、下发成功、补发完成。
- 警告状态使用橙色，表示缓存堆积、性能偏高、配置待下发。
- 错误状态使用红色，表示网关离线、下发失败、数据上报异常。
- 背景采用浅灰或白色，表格与卡片保持清晰边界，避免暗色大屏风格影响后台操作效率。
